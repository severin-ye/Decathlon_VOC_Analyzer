# 工作流编排与批处理模块

## 模块定位

本模块对应 `workflows/single_product_workflow.py` 和 `04_scripts/run_workflow.py` 等批处理入口。它把数据概览、标准化、索引构建和商品分析编排为一个可执行的单产品流水线。

论文中可将该模块定义为“实验与系统运行编排层”。它保证论文中的方法不是若干孤立组件，而是可以端到端运行、复用检查点并输出统一产物的系统。

## 主要创新点

1. LangGraph 状态机编排：系统使用 `StateGraph` 定义 `overview -> normalize -> index -> analyze` 的顺序，状态中显式保存输入参数和阶段产物。
2. 阶段可跳过与可复用：`skip_normalize`、`skip_index`、`reuse_extraction_artifact` 和 `reuse_analysis_checkpoint` 让实验可以复用中间产物，避免每次重复调用高成本模型。
3. 统一进度追踪：每个节点都会调用 `runtime_progress` 激活模块和步骤，使 CLI 或 API 运行时能够显示当前阶段、步骤和计数。
4. 单产品闭环：该工作流专门围绕一个 `category/product_id` 执行，适合论文案例分析、人工审核和消融实验。
5. 批处理配置隔离：README 中说明批处理入口默认可将 Qdrant 路径切到独立 run 目录，避免多个真实运行共享 store 时产生文件锁或结果污染。

## 具体实现

`SingleProductWorkflowState` 定义了工作流状态，包括 category、product_id、max_reviews、top_k_per_route、questions_per_aspect、use_llm、skip 标志、reuse 标志以及四个阶段产物。

`overview_node()` 调用 `DatasetService.build_overview()` 统计数据集概览。`normalize_node()` 在未跳过时调用 `normalize_dataset()`，只标准化目标商品所在类目与商品。`index_node()` 调用 `IndexService.build_index()` 为目标商品构建图文证据索引。`analyze_node()` 调用 `ProductAnalysisService.analyze()`，执行评论抽取、问题生成、召回、报告生成和产物持久化。

工作流通过 `builder.add_edge()` 固定阶段顺序，并用 `InMemorySaver` 作为 LangGraph checkpointer。虽然当前 checkpointer 是内存级，但业务层还额外支持 extraction artifact 和 analysis checkpoint 的磁盘复用。

## 与文献思路的关系

LangGraph 和 LangChain 的核心价值在于将 LLM 应用拆成可观测、可组合的状态节点，而不是把所有逻辑压入单次 prompt [1]。本项目借鉴这一工程思想，将 VOC 分析拆成标准化、索引和分析三个可监控阶段。

ReAct 强调推理与行动之间的结构化交替 [2]。本工作流虽然不是传统 agent，但同样体现了“先准备证据、再执行检索、最后生成结论”的行动序列。

## 可写入论文的表述

本文将完整 VOC 分析过程实现为单产品状态机流水线。该流水线依次执行数据概览、证据标准化、商品图文索引和证据约束报告生成。通过阶段跳过、中间产物复用和统一进度追踪，系统既支持交互式 API 使用，也支持论文实验中的批量运行与复现实验。
