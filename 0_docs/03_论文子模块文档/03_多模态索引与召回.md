# 多模态索引与召回模块

## 模块定位

本模块对应 `stage3_retrieval` 中的 `index_service.py`、`index_backends.py`、`embedding_service.py` 和 `retrieval_service.py`。它将标准化商品证据包转为可检索索引，并针对问题规划模块生成的 query 执行文本与图像双路召回。

论文中可将该模块定义为“商品侧证据检索层”。其目标不是回答用户问题，而是为每个评论方面找到可核验的商品文案证据和商品图像证据。

## 主要创新点

1. 评论方面驱动的商品图文召回：系统不是对商品做泛化摘要，而是由评论方面生成具体检索问题，再分别召回文本证据和图像证据。
2. 文本与图像分路索引：`IndexedEvidence.route` 明确区分 `text` 和 `image`。文本证据使用商品标题、描述、类目等文本块；图像证据包括整图和默认区域。
3. 原生视觉 embedding 与文本代理兼容：当 `image_embedding_backend=clip` 时，系统使用 CLIP 对整图或裁剪区域编码；否则退回到图像代理文本 embedding。该设计兼顾真实多模态能力和离线可运行性。
4. 本地 JSON 与 Qdrant 双后端：API 默认可使用本地 JSON 索引，批处理可切换到 Qdrant。两者共享 `IndexBackend` 抽象，便于在论文实验中比较不同后端或运行模式。
5. 语言均衡候选池：`RetrievalService` 在粗召回后按 route 与 language 构建候选池，避免单一语言证据垄断最终候选。
6. 按需索引构建：当检索目标商品尚未出现在索引中时，`get_or_create_product_snapshot()` 会自动为该商品构建索引并持久化，降低使用门槛。

## 具体实现

`IndexService.build_index()` 首先选择商品目录，加载 `ProductEvidencePackage`，然后调用 `_build_evidence_for_package()` 生成 `IndexedEvidence`。文本块被直接转换为 `route=text` 的证据，向量来自 `EmbeddingService.embed_text()`。图片被转换为 `route=image` 的证据，向量来自 `embed_image()`；若是图片区域，则传入 `crop_box` 对局部区域裁剪后编码。

`EmbeddingService` 支持多种向量链路。文本 embedding 可使用 DashScope OpenAI 兼容 API 的 `text-embedding-v4`，也可使用本地 Qwen3 embedding；当外部能力不可用且策略允许降级时，系统使用 64 维 hash embedding。图像 embedding 默认使用 `openai/clip-vit-base-patch32`，通过 `CLIPModel.get_image_features()` 和 `get_text_features()` 将图像与图像查询映射到同一空间。

`LocalIndexBackend` 将所有 `ProductIndexSnapshot` 写入 `local_evidence_index.json`，检索时对 query vector 与 evidence vector 做余弦相似度近似的点积计算。`QdrantIndexBackend` 则分别为文本和图像创建 collection，使用 Qdrant 的向量检索能力，并通过 `product_id`、`category_slug` 和 `route` 做过滤。

`RetrievalService.retrieve_for_package()` 对每个 `RetrievalQuestion` 执行召回。它先调用 `IndexService.search()`，每个 route 过采样 `top_k_per_route * 6`，再调用 `_build_language_balanced_candidate_pool()` 形成候选池。之后重排模块会进一步排序，最终 `_select_hits_with_route_coverage()` 保证文本和图像路线都尽量有证据进入最终结果。

## 与文献思路的关系

RAG 的基本思想是在生成前检索外部证据，以降低模型对参数记忆的依赖 [1]。本项目将该思想迁移到电商 VOC 场景，检索对象不是百科段落，而是商品页面中的文案、图片和图片区域。

CLIP 证明了图像与文本可通过自然语言监督映射到共同表征空间 [2]。本项目利用这一思路实现评论问题到商品图像的跨模态召回。

多阶段检索研究与 Qdrant 工程实践都强调先用 embedding 粗召回，再在较小候选集上重排 [3][4]。本模块的过采样、route 分层和后置 reranker 正是这一思想的工程化实现。

## 可写入论文的表述

本文构建了面向商品证据的双路多模态召回机制。对于每个由评论方面生成的问题，系统分别在商品文本索引和商品图像索引中进行候选召回，并通过过采样和语言均衡策略保留多样化证据。该设计避免了只依赖评论文本进行 VOC 总结的问题，使最终报告能够引用商品页面中的可核验证据。
