# 报告生成证据归因与回放模块

## 模块定位

本模块对应 `stage4_generation/analysis_service.py`、`report_refinement_service.py`、`artifact_sidecar_service.py`、`html_export_service.py` 和相关报告 schema。它串联评论抽取、问题规划、证据检索、质量评估、报告生成、证据归因、反馈回放和产物落盘。

论文中可将该模块定义为“证据约束生成层”。它不是简单调用大模型生成总结，而是在生成前构造 aspect 聚合、retrieval records 和 evidence nodes，在生成后进行结构校正、归因和回放处理。

## 主要创新点

1. 证据约束的报告生成：LLM 只在方面聚合和检索结果基础上生成报告，最终报告结构包括 strengths、weaknesses、controversies、evidence_gaps 和 suggestions。
2. 启发式生成兜底：当 LLM 不可用或运行策略允许降级时，系统使用 `_build_report_heuristic()` 生成完整报告，保证端到端流程可测试。
3. 报告后处理校准：`ReportRefinementService` 会去重 insight 和 suggestion，根据支持证据类型重新判定 owner 与 evidence_level，并修正“声称有图像支持但实际没有图像证据”的摘要。
4. Claim 级证据归因：系统构建 `claim_attributions`，把报告中的主张映射到 review evidence、product text evidence 和 product image evidence，支持 supported、partial、unsupported、contradicted 等状态。
5. 证据缺口建模：当检索无法提供足够商品证据时，报告不会强行确认评论观点，而是输出 evidence gap。这是降低生成幻觉的重要机制。
6. 回放与反馈侧边车：系统可读取 `feedback` 和 `replay` payload，在后续分析中复用人工反馈或持久问题，形成分析结果的迭代闭环。
7. 过程 trace：最终 response 中保留从抽取、问题、检索到报告和回放的 `ProcessTraceItem`，便于论文中的可解释性展示。

## 具体实现

`ProductAnalysisService.analyze()` 是核心入口。它首先加载商品证据包，然后通过 `_resolve_extraction()` 获取评论方面。若请求 `reuse_extraction_artifact=true`，系统复用已经落盘的 aspect 产物；否则重新抽取。

接着 `_resolve_question_retrieval_state()` 生成问题并执行检索。如果请求 `reuse_analysis_checkpoint=true`，系统会复用已落盘的问题、检索和质量评估结果；否则调用 `QuestionGenerationService` 和 `RetrievalService`，再计算检索质量和运行 profile。

系统随后调用 `_aggregate_aspects()` 对方面进行统计，形成每个 aspect 的正负中性数量、平均置信度、支持评论和证据片段。报告生成阶段优先使用 LLM，失败或禁用时使用 heuristic。报告生成后，`ReportRefinementService.refine()` 会执行去重、owner 分类、证据级别校准和建议修正。

在报告结构补全阶段，系统会生成 product impressions、customer impressions、aspect relations、supporting aspects、supporting reviews、supporting review evidence 和 supporting product evidence。然后 `_apply_replay_context()` 合并历史反馈或回放上下文。

最后 `_attach_claim_attribution()` 构建 claim 级证据归因，`_build_process_trace()` 构建过程追踪，`_persist_report()` 将分析 JSON、feedback sidecar、replay sidecar 等产物写入输出目录。

## 与文献思路的关系

RAG 的核心目标是让生成建立在外部证据之上 [1]。本模块进一步将证据约束从输入阶段延伸到输出阶段：不仅检索证据，还检查报告主张是否能被证据支撑。

RARR 等研究提出在生成后检索、修订和校正模型输出，以减少不实陈述 [2]。本项目的 report refinement、claim attribution 和 replay context 与这一思想一致，但落在商品 VOC 场景中，并把输出对象固定为可评估 schema。

结构化输出研究强调，LLM 在复杂 JSON 生成中可能出现字段缺失或格式不稳定 [3]。本项目通过 Pydantic schema、后处理补全和启发式兜底提升报告产物稳定性。

## 可写入论文的表述

本文设计了证据约束的 VOC 报告生成层。系统在生成前聚合方面级评论信号和多模态检索证据，在生成后执行结构补全、证据级别校准和 claim 级归因。最终报告中的每个主要洞察都可以追踪到用户评论、商品文本或商品图片证据；当证据不足时，系统显式输出证据缺口，而不是生成未经支持的结论。
