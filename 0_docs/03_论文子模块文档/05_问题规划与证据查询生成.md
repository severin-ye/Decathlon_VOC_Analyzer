# 问题规划与证据查询生成模块

## 模块定位

本模块对应 `stage4_generation/question_service.py` 和 `question_cache_service.py`。它的任务是把评论方面 `ReviewAspect` 转化为具体、可检索、带证据路线约束的问题 `RetrievalQuestion`。

论文中可将该模块定义为“VOC 到证据检索的桥接层”。评论方面本身通常是用户主观表达，例如“容量小”“材质偏硬”“性价比高”。系统不会直接用这些短标签检索商品证据，而是先规划问题意图，再生成检索问题。

## 主要创新点

1. 方面驱动的问题意图规划：系统为每个 aspect 生成固定意图序列，包括 `explicit_support`、`visual_confirmation`、`cross_modal_resolution`、`spec_check` 和 `visual_detail`。这些意图区分了文本证据、图像证据和图文联合证据的不同用途。
2. 检索路线显式约束：每个问题都带有 `expected_evidence_routes`，例如只查文本、只查图像或同时查文本与图像。召回层据此执行 route-aware retrieval。
3. 禁止概念与具体性边界：`QuestionIntent` 中包含 `forbidden_concepts` 和 `specificity_bound`，用于抑制 LLM 生成过度泛化、不可核验或引入外部知识的问题。
4. LLM 与启发式双模式：LLM 模式调用 Qwen 生成更自然的问题；启发式模式使用模板生成可检索问题。两者都经过 `_validate_question_candidate()` 标准化。
5. 问题缓存绑定 prompt 与 aspect digest：缓存签名包含 prompt variant、prompt digest、aspect digest、问题数量、模型与 API 配置，避免提示词或方面输入变化后误用旧问题。

## 具体实现

`generate_questions()` 首先为输入 aspects 构建缓存签名。如果缓存命中，则直接返回历史问题、意图和 warning。未命中时，系统调用 `plan_question_intents()` 为每个 aspect 规划意图，再根据运行策略决定使用 LLM 还是 heuristic。

`_build_intents_for_aspect()` 定义了五类问题意图。`explicit_support` 关注商品页文本是否直接支持评论观点；`visual_confirmation` 关注商品图片是否能提供结构或外观证据；`cross_modal_resolution` 要求综合图文判断是真实产品问题还是预期偏差；`spec_check` 收紧到规格、保修或明确声明；`visual_detail` 聚焦局部视觉细节。

LLM 模式下，系统将 aspect、opinion、evidence_span、usage_scene、sentiment、intent_type、expected_evidence_routes、forbidden_concepts 和 specificity_bound 作为 payload 输入提示词。返回问题后，系统验证 route 是否只包含 `text` 或 `image`，问题为空时退回启发式模板，并执行去重。

启发式模式下，系统用固定模板生成问题。例如中文模式中，`visual_confirmation` 会生成“商品图片中是否存在能支撑某方面判断的结构或外观证据？”。这些问题虽然不如 LLM 灵活，但足够支撑离线测试和回归验证。

## 与文献思路的关系

RAG 系统中的查询改写和问题分解通常用于提高检索命中率 [1]。本项目将该思想细化到 VOC 场景：不是对用户问题做泛化改写，而是把评论方面拆成多个证据需求。

ReAct 等工作强调推理与行动的交替，即模型不应一次性给出结论，而应先产生中间行动或查询 [2]。本模块体现了类似思想：评论方面先转为检索问题，检索完成后才进入报告生成。

多模态 RAG 强调不同模态需要不同检索策略 [3]。本模块通过 `expected_evidence_routes` 在问题阶段就声明证据路线，使后续召回天然支持文本、图像和图文联合验证。

## 可写入论文的表述

本文提出方面到证据问题的中间规划层。对于每个评论方面，系统生成若干具有明确检索意图和模态路线的问题，使主观 VOC 表达被转化为可执行的商品证据查询。该设计降低了直接用 aspect 标签检索造成的语义过宽问题，也为最终报告中的证据覆盖和证据缺口分析提供了结构化基础。
