# 评论建模与抽样模块

## 模块定位

本模块对应 `stage2_review_modeling`，主要包括 `review_service.py`、`review_sampling_service.py` 和 `deduplication_service.py`。它负责把商品评论从原始自然语言转化为方面级 VOC 信号，即 `ReviewAspect`。后续问题规划、证据检索和报告生成均以这些方面对象作为入口。

论文中可将该模块定义为“需求侧建模层”。商品证据包描述的是商品提供了什么，而评论方面描述的是用户关心什么、满意什么、抱怨什么。系统通过评论方面把 VOC 需求转化为后续检索问题。

## 主要创新点

1. 问题优先的评论抽样：默认 profile 为 `problem_first`，按照 1 星 30%、2 星 25%、3 星 20%、4 星 15%、5 星 10% 的目标配额抽样。当低星样本不足时，按照配置中的 fallback 顺序补位。该策略让有限预算优先覆盖高价值问题反馈。
2. LLM 与启发式双路径抽取：当 API Key 和运行策略允许时，系统调用 Qwen 进行结构化方面抽取；当 LLM 不可用或用户关闭 LLM 时，系统使用关键词、评分和情感提示词完成启发式抽取。该设计保证工程闭环可离线运行，也允许在满血模式下获得更强语义抽取能力。
3. 低信息评论过滤：系统会识别空评论、过短评论以及 `ok`、`good`、`great`、`ras` 等低信号短评，避免无意义评论稀释后续分析。
4. 方面级证据跨度保留：每个 `ReviewAspect` 不只保存方面名和情感，还保存 `opinion`、`evidence_span`、`usage_scene` 和 `confidence`。这使生成报告时能够回溯到原评论证据，而不是只统计抽象标签。
5. 抽取结果去重：`ReviewDeduplicationService` 以评论 ID、方面名、情感和证据文本为键去重，减少 LLM 重复输出或启发式重复匹配导致的噪声。

## 具体实现

`ReviewSamplingService.select_reviews()` 在 `max_reviews` 被设置时生效。系统首先按评分将评论放入 1 至 5 星桶和未评分桶，然后读取 `03_configs/review_sampling_profiles.json` 中的 active profile。目标数量通过权重乘以请求数量得到，余数按照小数部分和 fallback 优先级分配。如果某个星级评论不足，系统按 fallback 顺序将缺口重新分配给其他星级，最后生成 `ReviewSamplingPlan`，记录目标数、可用数、实际选择数、短缺数和补位数。

`ReviewExtractionService.extract()` 先调用 `_resolve_reviews()` 获取评论来源。用户既可以直接传入评论数组，也可以传入 `product_id` 让系统从标准化商品证据包中读取评论。随后 `_preprocess_review()` 清洗 HTML 与空白字符，判断评论是否 informative，并补齐语言提示。

结构化抽取有两条路径。LLM 路径调用 `QwenChatGateway.invoke_json()`，使用 `review_extraction_system` 提示词，并用 Pydantic schema 约束返回结构。启发式路径则使用 `ASPECT_KEYWORDS` 匹配容量、便携性、可用性、耐用性、价格价值、材料硬度等方面，用评分和负向词推断情感，并用关键词检测 travel、daily_use、outdoor_sports、summer 等使用场景。

抽取完成后，系统调用 `deduplicate()` 去重，并在 `persist_artifact=true` 时写入 `02_outputs/2_aspects/<category>/<product_id>.json`。产物中包含预处理评论、抽样计划、方面列表、跳过评论 ID 和 warning。

## 与文献思路的关系

方面级情感分析的核心思想是将评论从整句级情感拆分为面向属性的观点表达 [1]。本项目延续该思想，但不止输出 aspect-sentiment pair，而是将方面对象作为检索问题生成的上游输入。

LLM 结构化输出能力近年来被广泛用于信息抽取，但其可靠性依赖 schema 约束与错误处理 [2][3]。本模块通过 Pydantic schema、启发式回退和运行策略控制，把 LLM 抽取从一次性 prompt 调用变成可追踪、可降级的工程组件。

## 可写入论文的表述

本文将用户评论建模为方面级 VOC 单元。每个单元包含方面名称、情感极性、观点摘要、原文证据片段、使用场景和置信度。与传统评论汇总方法不同，本文不直接让语言模型总结所有评论，而是先构造可检索的需求侧对象，再由这些对象驱动商品侧证据检索。这一设计将“用户说了什么”与“商品证据能否支持”显式分离。
