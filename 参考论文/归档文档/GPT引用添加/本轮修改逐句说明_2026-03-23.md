### 2. 补写 Discharge Me 共享任务，强化“出院小结生成”的文献基础

原句：

> 现有研究表明,医疗文本生成中的幻觉具有明显的任务依赖性,目前并不存在一个统一的“医疗LLM幻觉率”指标。在临床笔记摘要任务中,对12,999个句子的人工作标显示句子级幻觉率为1.47%,遗漏率为3.45%,且44%的幻觉被判定为可能影响诊疗的重大错误 <sup>[32]</sup>。在出院小结生成任务中,研究通常不直接报告单一“幻觉率”,而是比较每份文书的错误数量; 一项JAMA Internal Medicine研究发现,LLM生成摘要平均包含2.91个独特错误,高于医生撰写版本的1.82个 <sup>[33]</sup>。在放射学报告生成中,问题更为突出:一项系统综述显示,纳入的15项研究均报告了不同程度的hallucinations、misdiagnoses或报告不一致问题 <sup>[34]</sup>; RadFlag进一步指出,高性能放射报告生成模型中约40%的生成句子可能包含幻觉 <sup>[35]</sup>。更严重的是,这些错误往往伴随较高置信度输出,使其在临床环境中特别危险 <sup>[10, 11]</sup>。现实世界的事件已记录了AI生成的医疗建议包含危险幻觉的案例,凸显了对可靠检测机制的迫切需求。

改后句子：

3. 这样修改后，任务依赖性这一论点更完整：临床摘要有 [32]，出院小结有 [33] 与 shared task [48-50]，放射学报告有 [34][35]，三类任务形成更清楚的横向对照。

### 3. 收紧 L2 的表述，避免过度概括

改后句子：

> **L2：缺失患者特定上下文。**大多数研究仍在通用医学问题、跨患者汇总语料或通用临床生成任务上评估幻觉,而没有把来自电子健康档案（EHR）的**个体化患者数据**直接组织为可核验的事实依据 <sup>[12, 13]</sup>。例如，检测"患者服用阿司匹林"是否为幻觉，需要了解患者的过敏史和当前用药情况------而现有基准测试通常缺乏这种患者级上下文。

1. 原句说“大多数研究在通用医学问题上评估幻觉”，这在方向上大体成立，但对已有临床文本生成、摘要、出院小结等工作来说太窄了。

3. 同时把“没有利用 EHR 个体化患者数据”改为“没有把 EHR 个体化患者数据组织为可核验的事实依据”，表述更精确，因为问题不只是“有没有数据”，而是“有没有把数据变成核验机制”。

1. [12][13] 支撑的是“从 EHR 构建医疗知识图谱/图结构表示是可行的”。

### 4. 收紧 L3 的表述，避免“当前数据集仅提供二元标签”这种过强断言

原句：

> **L3：证据支撑不足。**当前的数据集仅提供二元标签（幻觉 vs 事实），而没有解释**为什么**某个陈述是错误的，或者**什么证据**与其矛盾 <sup>[14]</sup>。这限制了可解释检测方法的发展。

改后句子：

> **L3：证据支撑不足。**现有医疗幻觉研究大多报告题目级正确率、句子级幻觉率或文书级错误负担，但较少提供面向患者级核验的结构化解释，说明**为什么**某个陈述是错误的，或者**什么证据**与其矛盾 <sup>[14, 32, 33]</sup>。这限制了可解释检测方法的发展。

为什么这样改：

1. 原句“当前的数据集仅提供二元标签”过于绝对，容易被单个反例推翻。
2. [14] 是 explainability 需求论文，它能支撑“临床场景需要解释”，但不能单独证明“所有当前数据集都只有二元标签”。
3. 改后使用“大多报告……但较少提供……”这种更稳健的说法，既保留论文的批评点，又更符合已核验文献的真实状态。

引用为什么这样改：

1. 保留 [14]，因为它支撑“为什么需要解释”。
2. 新增 [32][33]，因为这两篇分别代表句子级摘要安全评估和文书级错误统计，它们正好对应“目前常见报告口径是什么”。
3. 这样引用链条就从“只说需要解释”升级为“现有研究常见怎么报告 + 为什么这还不够解释性”。

### 10. 修复 Meditron-7B 案例研究中的关键引用错配

原句：

> 为了验证CuraView在真实场景中的适用性，我们将检测智能体应用于Meditron-7B <sup>[30]</sup>生成的患者discharge summary。Meditron-7B是基于Llama2-7B在PubMed摘要和临床指南上微调的开源医疗LLM，在医学问答任务上表现优异。我们使用Meditron-7B为25名MIMIC-IV患者生成了discharge summary，然后使用CuraView的检测智能体对这些生成文本进行幻觉检测。

改后句子：

> 为了验证CuraView在真实场景中的适用性，我们将检测智能体应用于基于 Meditron-7B 生成的患者 discharge summary。之所以选择该模型，一方面是因为 MEDITRON 系列代表了面向医学语料继续预训练的开源医疗 LLM 路线 <sup>[30]</sup>，另一方面是因为在 Discharge Me 共享任务中，已经出现了基于 Meditron-7B 的出院小结系统 MEDISCHARGE，证明该模型族可被直接用于 Brief Hospital Course 与 Discharge Instructions 生成 <sup>[49]</sup>。我们据此使用 Meditron-7B 为25名 MIMIC-IV 患者生成 discharge summary，然后使用 CuraView 的检测智能体对这些生成文本进行幻觉检测。

为什么这样改：

1. 原句把 [30] 直接挂在“用 Meditron-7B 生成出院小结”这个具体场景上，存在引用错配风险。
2. [30] 是 MEDITRON 系列模型背景论文，能支撑“这是一条面向医学预训练的模型路线”，但不能单独充分支撑“它已用于 Discharge Me 任务中的出院小结生成系统”。
3. 因此我把一句话拆成两个论点：
   第一层：为什么选这个模型族，用 [30]；
   第二层：为什么它和出院小结场景相关，用 [49]。
4. 这样改以后，模型背景和任务应用不再混为一谈。

引用为什么这样改：

1. [30] 保留，但只支撑“MEDITRON 是医学继续预训练路线”。
2. 新增 [49]，用来支撑“基于 Meditron-7B 的 MEDISCHARGE 已被用于 Discharge Me 出院小结任务”。
3. 这是本轮最关键的一处“错配修复型”引文修改。

## 十一、补写“agent 场景下小模型可优于大模型”的相关工作

### 19. 在多智能体框架部分补入“小模型在 agent 子任务中的优势”文献

新增句子：

> 近期 agent 文献还表明，参数规模并不是工具型智能体性能的唯一决定因素。在 function calling、多轮 tool use 与 workflow orchestration 等结构化子任务上，经过高质量 agent 数据合成、监督微调或规则奖励强化学习优化的小模型，可以达到与更大通用模型相当、甚至更优的表现 <sup>[51, 52, 53]</sup>。例如，**ToolACE** 说明仅 8B 参数的 function-calling 模型即可在 Berkeley Function-Calling Leaderboard 上与最新 GPT-4 级系统竞争 <sup>[51]</sup>；**APIGen-MT** 进一步报告，在多轮 agent 基准上，较小模型尤其可能超过更大的对手 <sup>[52]</sup>；**Nemotron-Research-Tool-N1** 则显示，基于规则奖励强化学习训练的 7B/14B 工具调用模型可在主要 benchmark 上超过 GPT-4o <sup>[53]</sup>。这些结果说明，在 agent 场景中，任务特化、结构化接口和训练数据质量在某些子任务上可以比单纯扩大参数规模更关键。

为什么这样改：

1. 这一轮用户明确要求把“agent 场景下小模型或微调小模型可能优于更大模型”这一重要观点写进正文。
2. 这条论点更适合放在“多智能体系统与 LLM 应用”部分，而不是放在医疗出院小结任务谱系里，因为它讨论的是 agent 训练与工具调用能力，而不是 Discharge Me 本身。
3. 我把句子写成“在特定结构化子任务上成立”的限定性表述，避免把它误写成无条件的一般规律。

引用为什么这样改：

1. [51] 用于支撑 8B function-calling 模型可以与 GPT-4 级系统竞争。
2. [52] 用于支撑在多轮 agent setting 下，小模型可能超过更大对手。
3. [53] 用于支撑经规则奖励强化学习训练的 7B/14B 工具调用模型可超过 GPT-4o。
4. 这三篇合在一起，分别覆盖了 function calling、多轮 agent、RL tool use 三类子场景，使这条观点不依赖单篇论文。
