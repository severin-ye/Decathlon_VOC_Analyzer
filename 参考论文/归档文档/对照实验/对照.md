RAGTruth 与 QAGS 可落地复现手册（逻辑版）
第一部分：RAGTruth
1. 这个方法到底在做什么

RAGTruth 的核心思想不是“直接判断整段回答对不对”，而是：

先把回答拆成一个个可验证的小事实
再为每个小事实找证据
再判断这个小事实是否被证据支持
最后把所有小事实的判断结果汇总，得到整段回答的 hallucination 情况

所以它本质上是一个：

回答 → claim拆分 → 证据检索 → claim级验证 → 汇总

的检测框架。

2. 适用输入与输出
输入

RAGTruth 这一类方法的输入，逻辑上包括三部分：

1. question
2. answer
3. 与 answer 相关的可用文档 / 检索语料 / context

其中最关键的是：

answer 是待检测对象
context / document 是验证依据
输出

输出不是只给一个“真 / 假”，而是两层：

第一层：claim-level 输出

对于回答中的每个 claim，给出类似下面的结果：

- supported
- unsupported
- contradictory
第二层：answer-level 输出

把所有 claim 的结果综合起来，得到：

- 整体 hallucination 程度
- 整体 factual consistency 情况
3. 完整 pipeline
Step 0：准备待检测输入

这一阶段没有复杂逻辑，就是明确系统要处理什么。

输入
question
answer
context / document collection
输出
一个待验证样本：
(question, answer, context)
Step 1：把 answer 拆成 atomic claims
这一步在逻辑上要干什么

这一步的目标是：

把一个长回答拆成多个“最小可验证事实单元”。

因为如果不拆，你只能对整段回答做粗判断。
而一旦拆成 claim，就可以知道：

到底哪一句有问题
是哪一种问题
是无证据，还是和证据矛盾
这里为什么必须拆

例如回答是：

The patient has diabetes and hypertension.

如果不拆，这是一整句。
但实际上它包含了两个独立事实：

Claim 1: The patient has diabetes.
Claim 2: The patient has hypertension.

这两个事实是可以分别成立、分别出错的。

比如真实文档可能是：

Patient diagnosed with diabetes.
No history of hypertension.

那么最后结果就会变成：

Claim 1 正确
Claim 2 错误

这正是 RAGTruth 想要的粒度。

这一阶段的输入
answer
这一阶段的处理

这一阶段由 Qwen Plus API 完成：

读取 answer
找出其中所有可验证事实
将其拆成 atomic claims
这一阶段的输出
claims = [c1, c2, c3, ...]

例如：

answer:
The patient has diabetes and hypertension.

claims:
1. The patient has diabetes.
2. The patient has hypertension.
这一阶段最容易误解的点
误解 1：claim 不是句子切分

不是把 answer 按句号切开就结束了。
一个句子里可能有多个事实，一个事实也可能只占半句。

误解 2：claim 不是关键词抽取

claim 必须是一个完整事实，不是单个实体或短语。

错误例子：

diabetes
hypertension
patient

这些都不能直接验证。

误解 3：claim 不能拆得过细

如果拆成词级或属性碎片，会失去“事实”意义，后续无法验证。

Step 2：为每个 claim 找证据
这一步在逻辑上要干什么

对于每个 claim，需要从 context 或文档集合里找到最相关的证据片段。
因为后面不能凭空判断，必须让验证建立在证据之上。

所以这一阶段的逻辑是：

claim → 找相关证据
为什么不能直接用整段文档判断

因为原始文档可能很长，包含很多无关内容。
如果把整个文档直接扔给判断模块：

容易被噪声干扰
容易漏掉关键句
容易让模型凭常识而不是凭证据判断

因此必须先缩小到与 claim 最相关的 evidence。

输入
claim
context / document collection
处理

对每个 claim：

以 claim 为中心构造检索需求
在文档集合中查找与其最相关的片段
得到若干 candidate evidence
输出
claim_i -> evidence_i

例如：

Claim:
The patient has hypertension.

Retrieved evidence:
- No history of hypertension.
- Blood pressure remained normal during hospitalization.
这一阶段最容易误解的点
误解 1：检索对象应该是 claim，不是整个 answer

RAGTruth 的关键点之一就是claim-level grounding。
如果你拿整个 answer 去检索，会把多个事实混在一起，导致后面验证混乱。

误解 2：retrieved evidence 不是最终答案，只是候选依据

这一步的任务不是判定，而是“找材料”。

误解 3：证据可能支持，也可能反驳

很多人以为 evidence 只是“支持证据”。
其实不是。
能明确否定 claim 的片段，同样是重要 evidence，而且在医疗等高风险场景更关键。

Step 3：做 claim–evidence verification
这一步在逻辑上要干什么

这是整个方法最核心的一步。

目标是判断：

这个 claim 是否被证据支持？

判断结果一般落在三类里：

1. supported
2. unsupported
3. contradictory
三类标签分别是什么意思
supported

证据明确支持 claim

例如：

Claim:
The patient has diabetes.

Evidence:
Patient diagnosed with diabetes mellitus in 2019.

这就是 supported。

unsupported

证据里没有足够信息支持这个 claim
注意，这不等于 claim 一定错了，而是：

从当前证据看，无法证实它。

contradictory

证据明确与 claim 冲突

例如：

Claim:
The patient has hypertension.

Evidence:
No history of hypertension.

这是明显矛盾。

输入
claim
retrieved evidence
处理

这一阶段由 Qwen Plus API 完成。

它要做的逻辑不是“自由回答”，而是：

读取 claim
读取对应 evidence
只基于 evidence 判断三分类标签
输出该 claim 的验证结果
输出
claim_i -> label_i

例如：

Claim 1 -> supported
Claim 2 -> contradictory
Claim 3 -> unsupported
这一阶段最容易误解的点
误解 1：unsupported 和 contradictory 不是一回事

这是最容易混淆的地方。

unsupported：证据里没有足够支持
contradictory：证据明确说相反的话

这两个错误等级完全不同。

误解 2：判断必须绑定 evidence

这一步的逻辑核心是 evidence-grounded。
不是让模型根据常识猜，而是让它根据 retrieved evidence 判定。

误解 3：一个 claim 可能对应多条 evidence

因此最终判定不是简单看某一句，而是综合 evidence 集合来判断。

Step 4：把 claim-level 结果汇总成 answer-level 结论
这一步在逻辑上要干什么

因为最终通常还是要知道：

这整段 answer 到底有多严重的 hallucination？

所以必须把多个 claim 的结果合成整体结果。

输入
[c1_label, c2_label, c3_label, ...]
处理

统计所有 claim 的验证结果，例如：

supported 有多少个
unsupported 有多少个
contradictory 有多少个

然后形成 answer-level 的总体结论。

输出

输出可以是两种形式：

形式 1：整体标签
This answer contains hallucination.
形式 2：整体分数
Hallucination score = ...
这一阶段最容易误解的点
误解 1：不是“只要有一个错就整段全错”

这取决于你的实验定义。
RAGTruth 的价值恰恰在于它允许你保留细粒度信息，而不必把所有错误都压成一个粗标签。

误解 2：claim-level 信息不能丢

在实验中，最有价值的往往不是最后总分，而是：

哪个 claim 错了
错误类型是什么
是否出现 contradiction
4. RAGTruth 的最小可落地复现单元

如果你想最小化地复现这个框架，那么你至少要保留下面四个模块：

1. Claim decomposition
2. Claim-based retrieval
3. Claim–evidence verification
4. Aggregation

其中只有两部分需要 LLM：

- claim decomposition → Qwen Plus
- claim–evidence verification → Qwen Plus

其余部分保持普通流程逻辑即可。

5. RAGTruth 一句话本质
把回答拆成小事实，然后逐条做“证据约束下的事实核查”。
第二部分：QAGS
1. 这个方法到底在做什么

QAGS 的核心思想和 RAGTruth 不一样。

它不是：

summary → claim → evidence matching

而是：

summary → 生成问题 → 分别用 summary 和 source 回答 → 比较答案是否一致

所以它的本质是：

用问答一致性来检测 summary 是否忠于 source document。

2. 适用输入与输出
输入

QAGS 的典型输入是：

1. source document
2. summary

其中：

document 是原文
summary 是待验证摘要
输出

输出是 summary 的 factual consistency 情况，通常可以细到问题级，也可以汇总为总分。

3. 完整 pipeline
Step 0：准备 document 与 summary
输入
document
summary
输出
一个待检测样本：
(document, summary)
Step 1：从 summary 中生成可验证问题
这一步在逻辑上要干什么

QAGS 不直接去判断“summary 这句话对不对”，而是先把 summary 中的重要事实变成问题。

也就是说，它把“事实验证”改写成了“问答一致性验证”。

为什么要这样做

因为如果一个 summary 的事实是真的，那么：

从 summary 中提出的问题
应该既能被 summary 回答
也能被原始 document 回答
而且两个答案应该一致

如果两边答案不一致，就说明 summary 中的事实很可能偏离了 source。

输入
summary
处理

这一阶段由 Qwen Plus API 完成。

它需要做的是：

阅读 summary
找出其中值得验证的事实
将这些事实转换成问题
输出
questions = [q1, q2, q3, ...]

例如：

Summary:
John became champion in 2021.

Questions:
1. When did John become champion?
这一阶段最容易误解的点
误解 1：不是任意生成问题

生成的问题必须和 summary 中的事实紧密对应。
如果问题偏离 summary 的关键事实，后面比较就没有意义。

误解 2：问题的作用不是“补充信息”，而是“检查事实”

QAGS 的 question generation 是验证的一部分，不是开放式问答任务。

误解 3：问题粒度决定验证粒度

如果问题太泛，就难以定位错误；
如果问题太细碎，又会让比较变得噪声很大。

Step 2：分别用 summary 和 document 回答这些问题
这一步在逻辑上要干什么

对于每一个问题，需要做两次问答：

Q + summary  → answer_from_summary
Q + document → answer_from_document

这样你就得到两个答案，然后才能检查它们是否一致。

输入
question
summary
document
处理

这一步同样由 Qwen Plus API 完成，但要分两次：

第一次

基于 summary 回答问题

第二次

基于 document 回答同一个问题

输出
(question, answer_summ, answer_doc)

例如：

Question:
When did John become champion?

Answer from summary:
2021

Answer from document:
2020
这一步为什么关键

QAGS 的核心假设就是：

如果 summary 是 factual 的，那么针对同一个问题，summary 与 source document 应该导出一致答案。

所以真正要比的不是“句子表面像不像”，而是：

它们是否对同一个事实给出相同答案
这一阶段最容易误解的点
误解 1：两次 QA 的语境不一样

一次是基于 summary，另一次是基于 document。
这一点不能混。

误解 2：QAGS 不是拿问题去问世界知识

它不是开放问答，不允许脱离给定上下文去回答。

误解 3：document 侧回答才是验证依据

summary 侧回答反映“摘要声称了什么”，
document 侧回答反映“原文真正说了什么”。

Step 3：比较两个答案是否一致
这一步在逻辑上要干什么

拿同一个问题的两个答案：

answer_from_summary
answer_from_document

判断它们是不是一致。

输入
(question, answer_summ, answer_doc)
处理

对每个问题：

比较 summary-answer 与 document-answer
若一致，则说明这一事实大概率被正确保留
若不一致，则说明 summary 中对应事实可能有 hallucination 或 distortion
输出
question_i -> consistent / inconsistent

例如：

Q1 -> inconsistent

因为：

summary says: 2021
document says: 2020
这一阶段最容易误解的点
误解 1：QAGS 检查的是“答案一致性”，不是“句子相似度”

这是最核心的一点。

它不是问：

summary 和 document 写得像不像

而是问：

面对同一个事实问题，它们给出的答案是不是同一个
误解 2：不一致不一定意味着 summary 完全虚构

它也可能是：

时间错了
数字错了
人名错了
因果关系错了

所以 QAGS 更准确地说是在检测事实偏移。

Step 4：把所有 QA 比较结果汇总为 summary-level 分数
这一步在逻辑上要干什么

因为 summary 通常包含多个事实，所以需要把多个 question-level 结果合成整体结论。

输入
[q1_result, q2_result, q3_result, ...]
处理

统计：

一致的问题有多少
不一致的问题有多少

然后形成 summary-level factual consistency score。

输出

例如：

Consistency score = 一致问题数 / 问题总数

或者：

This summary contains factual inconsistency.
这一阶段最容易误解的点
误解 1：QAGS 的最终分数依赖前面问题生成的质量

也就是说，QAGS 的表现不只取决于最后比较，还强烈依赖：

summary 中被抽到了哪些问题
误解 2：QAGS 只能检测“被问题覆盖到的事实”

如果 summary 中某个错误事实没有被转成问题，它就不会被检出。

这也是它和 claim-based 方法的重要区别之一。

4. QAGS 的最小可落地复现单元

如果只保留最核心逻辑，那么 QAGS 最少需要下面三个模块：

1. Question generation from summary
2. Dual QA on summary and document
3. Answer comparison + aggregation

其中需要 LLM 的部分只有：

- question generation → Qwen Plus
- QA on summary → Qwen Plus
- QA on document → Qwen Plus

其他部分就是结果组织与一致性比较逻辑。

5. QAGS 一句话本质
把摘要中的事实改写成问题，再检查“摘要答案”和“原文答案”是否一致。
第三部分：RAGTruth 与 QAGS 的根本差别
1. 验证单位不同
RAGTruth

验证单位是：

claim

也就是“小事实单元”。

QAGS

验证单位是：

question-answer pair

也就是“通过问题间接检查事实”。

2. 验证路径不同
RAGTruth
answer → claim → evidence → label
QAGS
summary → question → dual answers → consistency
3. 它们各自更适合什么
RAGTruth 更适合
回答检测
RAG 输出检测
医疗 / 法律 / 高风险场景
需要知道“哪条 claim 错了”的场景

因为它保留了很强的 claim-level 粒度。

QAGS 更适合
摘要一致性检测
用 QA 方式做 indirect verification 的任务
想把“事实对齐”转成问答一致性的问题设定
4. 两者最核心的 methodological difference

可以用一句话概括：

RAGTruth 是“直接验证事实”
QAGS 是“通过提问间接验证事实”

question, answer, context

Step 1:
Qwen Plus 把 answer 拆成 claims

Step 2:
对每个 claim 找 evidence

Step 3:
Qwen Plus 判断 claim 与 evidence 的关系

Step 4:
汇总 claim-level 结果
QAGS 最小骨架
输入：
document, summary

Step 1:
Qwen Plus 从 summary 生成 questions

Step 2:
Qwen Plus 基于 summary 回答问题
Qwen Plus 基于 document 回答问题

Step 3:
比较两边答案是否一致

Step 4:
汇总所有问题的结果
第五部分：你现在最应该注意的复现重点
对 RAGTruth 来说，最关键的是三件事
1. claim 拆分粒度是否合理
2. evidence 是否真的是对 claim 检索来的
3. verification 是否严格绑定 evidence
对 QAGS 来说，最关键的是三件事
1. 问题是否真正覆盖 summary 中的关键事实
2. 两次 QA 是否严格限制在各自上下文中
3. 最终比较的是“答案一致性”而不是“句子相似度”
