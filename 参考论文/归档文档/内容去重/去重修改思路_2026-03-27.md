# 论文去重修改思路（2026-03-27）

## 1. 文档目的

本文档不是直接修改正文，而是为后续正文精简提供一份逐步执行的操作说明。核心目标是：

1. 去除跨章节的重复表述。
2. 不删除论文中的实质内容。
3. 现有重要表述、定义、论点、机制说明，至少在全文中保留一次。
4. 通过“收束到唯一主位置 + 其他位置改为引用式简述”的方式，实现去重而不是减料。

换言之，本轮修改的原则不是“删内容”，而是“避免同一内容被完整讲两遍、三遍、四遍”。

## 2. 总体修改原则

### 2.1 保留原则

凡是以下内容，均不得从全文中消失，只能决定其“主保留位置”在哪里：

1. 七类医疗幻觉分类法。
2. E1-E4 证据分级定义。
3. L1-L4 研究局限性。
4. RQ1-RQ4 与 C1-C4 的问题设定。
5. 多智能体生成-检测框架的双重价值。
6. GraphRAG 的方法优势与领域定制价值。
7. 统一的结构化输出可靠性流水线。
8. 8B vs 14B 的工程可用性结论。
9. 主实验结果与关键数据口径。
10. 五项贡献。

### 2.2 去重原则

去重时不采用“简单删除”，而采用以下四种手段：

1. 主定义保留：把完整定义保留在最合适的章节。
2. 次级出现压缩：其他章节只保留一句概括，不再整段重讲。
3. 作用转换：把“定义性段落”改写为“本文采用……，详见第X节”的引用式表述。
4. 章节分工明确：引言负责提出问题，背景负责定义术语，方法负责解释机制，实验负责展示结果，讨论负责解释意义，结论负责压缩回收。

### 2.3 禁止事项

1. 不允许把某个概念从全文彻底删掉。
2. 不允许为了缩短篇幅而删去关键论证链。
3. 不允许把同一段信息拆散到多处后造成信息残缺。
4. 不允许把定义性内容删掉却只剩结论性断言。

## 3. 全文去重的执行顺序

建议严格按以下顺序修改，避免前面刚删、后面又补、最后结构混乱。

### 第一步：先确定每类内容的“唯一主保留位置”

先不要动文字，先做全局分配。建议如下：

1. 七类幻觉分类法：主保留在第2章。
2. E1-E4 证据分级：主保留在第2章。
3. L1-L4：主保留在第1章引言的问题提出部分。
4. RQ1-RQ4 / C1-C4：主保留在第1章引言。
5. 多智能体双重价值：主保留在第4章方法，结论只简短回收。
6. GraphRAG 优势：技术细节主保留在第3章相关工作与第4章方法，讨论只解释意义。
7. 结构化输出可靠性流水线：主保留在第4章方法。
8. 8B vs 14B 工程可用性：设置在第5章，结果在第6章，讨论只保留部署含义。
9. 主实验结果：主保留在第6章，摘要和结论保留高度压缩版。
10. 五项贡献：完整展开主保留在第1章，结论保留压缩版。

只有主位置确定后，后续每一章才知道自己该保留什么、压缩什么。

### 第二步：先处理定义性重复，再处理论证性重复，最后处理总结性重复

建议顺序如下：

1. 先改第1章和第2章之间的重复。
2. 再改第1章、第3章、第4章之间的机制重复。
3. 再改第5章、第6章、第7章之间的工程结论重复。
4. 最后改第8章结论中的回收性重复。

原因是：定义层去重会影响全篇引用方式；机制层去重会影响方法与讨论的边界；结论层去重必须放在最后，否则容易把前文尚未稳定的表达提前删乱。

## 4. 逐章修改思路

摘要目前信息很多，但它本身不算主要问题，因为摘要本来就允许浓缩重复正文关键点。这里不建议大删，只建议控制功能边界。

修改思路：

1. 保留七类幻觉分类法的列举，因为摘要需要交代创新点。

摘要可以“提到”所有核心贡献，但不要替代正文定义它们。

## 4.2 引言：从“完整讲一遍”改为“提出问题并概述答案”

文件：01_introduction.md

第1章是当前重复最密集的区域之一。问题主要在于：

1. 七类分类法先在安全威胁部分提了一次，后面贡献1又完整展开一次。
2. 医疗幻觉率任务依赖性的文献综述，和第2章高度相似。
3. 结构化输出流水线在贡献2、贡献4附近反复铺陈。
4. 多智能体双重价值写得过满，和第4章、第8章高度重合。

修改思路：

1. 在“医疗幻觉：关键的安全威胁”部分保留一句：我们识别出七类关键幻觉类型，详细定义见第2.2节。
2. 删除或压缩贡献1中对七类类型逐项展开定义的长句，只保留“提出首个临床文档导向的句子级分类法”。
3. 将“医疗幻觉率任务依赖性”的详细文献数据压缩为高层概括，把具体数字和任务分野保留给第2章。
4. 保留 L1-L4、RQ1-RQ4、C1-C4，因为这部分是引言的正职，不建议削弱。
5. 对“多智能体双重价值”的处理方式改为：引言中只保留价值主张，不展开数据字段和蒸馏用途细节。
6. 对“结构化输出可靠性流水线”的处理方式改为：引言中保留其作为贡献4的定位，但不再完整解释 schema 约束、生成侧约束、后验校验三者的全部细节。
7. 五项贡献要保留，但每项以“贡献是什么”为主，不以“机制如何实现”为主。

执行标准：

引言修改后，读者应清楚知道本文解决什么问题、为什么重要、贡献是什么；但不应在引言中已经看完第2章和第4章的大部分内容。

## 4.3 背景：承担唯一正式定义职责

执行标准：

## 4.4 相关工作：聚焦对比，不再重复自家方法细节

文件：03_related_work.md

第3章的任务是“讲别人做了什么，我们和他们差在哪里”，不是重新讲本文方法。

当前问题主要在于：

1. 在“与现有工作的比较”部分，已经开始较完整地重讲本文的系统贡献。
2. 尤其是结构化输出可靠性流水线，被写成了接近方法章的缩略版。

2. 将“结构化输出可靠性流水线”的段落改成差异说明，而不是机制说明。
3. 删除或压缩那些已经接近“本文方法介绍”的细节句，比如具体约束链条、解析修复、三层验证的完整罗列。

执行标准：

第3章改完后，读者应明白本文相较已有工作的新意，但真正想知道系统怎么做，仍需进入第4章。

## 4.5 方法：保留完整机制，但避免在方法内部重复定义

文件：04_methodology.md

第4章是方法主战场，允许细讲，但要避免“方法章内部自己又重复一遍背景章定义”。

当前问题主要在于：

1. 在生成智能体中又重新完整定义了一次 E1-E4。
2. “检测即数据生产”的双重价值和引言、结论高度相似。
3. 结构化输出流水线写得完整合理，但其价值主张与引言/讨论/结论措辞重叠很高。

修改思路：

1. 保留表1，因为它是方法实现层面的任务映射，不等同于背景章的纯定义。
2. 将“证据分级系统”改写为“本文在生成阶段按第2.3节所定义的 E1-E4 标注各类样本”，重点讲应用，不再逐级重定义。

4. 结构化输出可靠性流水线保留完整技术说明，因为它的主位置就在第4章。
5. 但方法章中要避免在不同小节反复重复同一价值判断，例如“不是能回答，而是能稳定进入链路”这类总结句，建议保留一次即可。

执行标准：

第4章应成为“唯一完整技术说明来源”，但不承担重新定义第2章术语的职责。

## 4.8 讨论：只解释意义，不再重述机制

文件：07_discussion.md

第7章当前的问题是：一部分段落在做结果讨论，另一部分段落在重新讲方法与工程机制。

修改思路：

1. 保留“微调重要性”“GraphRAG 的临床意义”“精确率-召回率权衡”“部署中的成本-性能权衡”等讨论内容。
2. 将结构化输出策略部分改写成结果含义式讨论，不再完整解释 schema 约束、类型校验、失败重试这些方法细节。
3. 将 8B/14B 差异压缩为部署结论：在相同结构化输出约束下，14B 展现出更好的端到端工程可用性。
4. GraphRAG 的段落重点谈其临床意义与可扩展性，而不是再次详细解释图结构检索本身。

执行标准：

第7章改完后，读者应获得“结果意味着什么”，而不是感觉自己又看了一遍方法章。

## 4.9 结论：改成真正的压缩回收，而不是第二篇引言

文件：08_conclusion.md

第8章目前最大的问题是：五项贡献虽是总结，但第2项和第4项展开过长，与第1章、第4章重复度明显偏高。

修改思路：

1. 保留五项贡献的框架。
2. 将每项贡献压缩为“是什么 + 为什么重要”，不再展开实现细节。
3. 对“多智能体双重价值”只保留一句总结，不再详细列出每条数据包含哪些字段。
4. 对“结构化输出可靠性流水线”只保留一句总结，不再重述 schema 约束、生成侧约束、后验校验三分法。
5. 保留主结果数字，因为结论必须回收关键结果。
6. 未来工作、广泛意义可以保留，但尽量避免再次与引言的价值主张一字多用。

执行标准：

结论应该像“压缩回收”，而不是“重新详细介绍本文贡献”。

## 5. 重点重复项的专门处理办法

## 5.1 七类幻觉分类法

处理原则：

1. 完整定义只留在第2章。
2. 第1章只简述“我们提出七类分类法，详见第2.2节”。
3. 第4章表1保留为“任务映射表”而不是再次承担理论定义职责。
4. 第8章只回收一句“提出七类分类法”。

这样做的结果是：

七类内容没有消失，但从多次完整定义变成“一次定义，多次引用”。

## 5.2 E1-E4 证据分级

处理原则：

1. 完整定义只留在第2章。
2. 第4章只写生成阶段如何使用 E1-E4。
3. 第5章只写评估阶段如何将 E4 作为主指标、E3+E4 作为补充指标。
4. 第6章表格与结果中只按指标解释，不再重定义。

## 5.3 多智能体双重价值

处理原则：

1. 最完整说明留在第4章。
2. 第1章只说这是系统价值之一。
3. 第8章只做一句浓缩回收。

## 5.4 结构化输出可靠性流水线

处理原则：

1. 第4章为唯一完整说明位置。
2. 第1章仅保留其作为贡献4的高层摘要。
3. 第3章只写这是与既有系统相比的工程差异点。
4. 第7章只解释其部署意义，不再重讲机制。
5. 第8章只保留一句总结。

## 5.5 8B vs 14B 工程可用性

处理原则：

1. 第5章讲实验设定与统一约束条件。
2. 第6章讲结果与观察。
3. 第7章讲部署含义。
4. 不在三章中同时完整重讲 JSON 准确率、字段名正确率、统一链路与结论。

## 6. 建议的具体修改节奏

如果后续进入实际改稿，建议分四轮完成。

### 第一轮：只处理第1章和第2章

目标：

1. 解决七类分类法重复。
2. 解决医疗幻觉率任务依赖性段落重复。
3. 解决 E1-E4 的主定义位置问题。

完成标准：

定义性内容全部收束到第2章，引言不再承担二次定义角色。

### 第二轮：处理第3章和第4章

目标：

1. 第3章停止重讲本文机制。
2. 第4章保留完整方法，但避免内部再定义第2章术语。
3. 收束“多智能体双重价值”和“结构化输出流水线”的重复句式。

完成标准：

相关工作负责对比，方法负责说明。

### 第三轮：处理第5章、第6章、第7章

目标：

1. 明确“设置-结果-意义”的章节分工。
2. 去掉 8B/14B 结论在三章中反复整段重述的问题。
3. 保留所有关键数字，但不重复讲相同工程故事。

完成标准：

同一组工程结论只在最合适的章节完整出现一次。

### 第四轮：处理第8章结论

目标：

1. 将结论压缩成真正总结。
2. 保留五项贡献，但去掉实现层细节复述。
3. 保留主结果与未来方向。

完成标准：

结论成为全文收束，而不是对前文的再次展开。

## 7. 最终验收标准

完成去重后，应满足以下标准：

1. 每个核心概念在全文中至少有一个完整定义或完整说明位置。
2. 其他章节再提及时，以概括和引用为主。
3. 任何关键观点都没有丢失。
4. 读者从第1章读到第8章时，不会连续两次看到几乎同构的大段解释。
5. 论文整体逻辑更加清晰：引言提问题，背景给定义，相关工作做对比，方法讲实现，实验给证据，讨论讲意义，结论做回收。

## 8. 一句话执行口径

本轮去重修改的操作口径可以概括为：

**不是删除内容，而是为每一类内容指定唯一主位置，并把其他位置改成必要的简述、引用和回收。**
