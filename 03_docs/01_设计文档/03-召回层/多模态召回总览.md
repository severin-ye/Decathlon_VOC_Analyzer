# 多模态召回总览

## 召回目标

本项目的召回目标不是“找最像的图片”或“找最相关的文本块”，而是找到最能解释评论观点的商品证据。

最终召回对象应当是证据包，而不是单一对象。

这里的关键变化是：

- 不直接拿评论原句去做召回
- 也不直接拿 aspect 标签去做召回
- 而是先从评论方面对象中生成一组“需要被澄清的问题”
- 再用这些问题作为召回单元去检索商品图像与商品文案

也就是说，召回的直接输入不再是评论本身，而是“问题化后的查询意图”。

## 项目中的召回主线

这里采用的不是纯文本 RAG，也不是纯视觉检索，而是“问题生成 + 双路召回 + 产品级合并”：

- 步骤 1：评论抽取出 aspect 级结构对象
- 步骤 2：针对每个 aspect 生成 k 个待澄清问题
- 路 1：围绕问题做商品图像召回
- 路 2：围绕问题做商品文案召回
- 路 3：按 product_id 合并证据并做 rerank

典型问题示例：

- 这条评论提到的“收纳分区多”在商品文案中有没有明确描述？
- 能否从商品图片中看到支持“可装护照和多张卡”的结构证据？
- 这条关于“偏硬”的反馈更可能对应材料、结构，还是使用预期偏差？

## 为什么这样设计

因为商品场景天然分为两类信息：

- 商品证据具有明显多模态属性。
- 评论证据主体是文本，但评论原句往往过于口语化、碎片化、情绪化，不适合作为稳定召回 query。

因此更合理的做法不是“评论直接召回”，而是“评论先转成结构化方面对象，再转成一组需要被验证的问题”。

这样做有三个直接收益：

- 召回目标更明确，减少口语化评论带来的噪声
- 文本路与图像路可以围绕同一问题对齐，而不是各自猜评论意图
- 后续生成阶段可以回答“这个问题被哪些证据解释了”，解释性更强

## 与通用多模态 RAG 的关系

对话中先讨论了多模态 RAG 的一般实现方式，之后再落到商品场景。对于当前项目，通用结论被收敛为：

- 检索阶段用多模态 embedding
- 检索前增加问题生成层，把评论意图改写为待澄清问题集合
- 必要时做 text route 与 image route 的双路召回
- 召回后做重排与证据包构造
- 最终交给生成模型做证据约束总结
