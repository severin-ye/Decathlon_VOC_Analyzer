## 1. RARR

### 原文核心

RARR 的核心是 **Retrofit Attribution using Research and Revision**，也就是对已经生成的回答进行后验归因：先自动寻找外部证据，再对没有证据支撑的内容做最小修改，同时尽量保留原回答。论文明确说它解决的是语言模型输出没有内置外部证据归因机制的问题，并且包含两个步骤：找 attribution、修订 unsupported content。([arXiv][1])

### 对你项目的价值

这个最适合你的项目。

因为你当前不是没有报告，而是报告生成后：

```text
strengths / weaknesses / suggestions
```

里面的句子不一定能稳定绑定到：

```text
review evidence
product evidence
image evidence
```

所以 RARR 应该作为 **P0 主引用**，不是普通参考。

### 建议修改报告写法

原报告可以更明确地写成：

```text
RARR 对当前项目最直接的启发不是重新训练生成模型，而是在 report generation 之后增加一层 post-hoc attribution and revision。具体来说，系统先生成候选报告，再把报告拆成 claim 级单元，对每个 claim 检索并绑定 review / product / image evidence；如果无法绑定，则将该 claim 降级、删除或转入 evidence_gap。
```

### 建议落地方案

新增一个服务：

```text
ClaimAttributionService
```

输入：

```json
{
  "report_section": "suggestions",
  "claim_text": "The sunglasses are suitable for long outdoor running.",
  "candidate_evidence": [
    "review_001",
    "review_009",
    "product_desc_002"
  ]
}
```

输出：

```json
{
  "claim_text": "The sunglasses are suitable for long outdoor running.",
  "support_status": "unsupported",
  "support_type": null,
  "support_ids": [],
  "revision_action": "downgrade",
  "revised_claim": "Current evidence is insufficient to confirm suitability for long outdoor running."
}
```

## 2. ALCE

### 原文核心

ALCE 不是一个普通 RAG 方法，而是一个 **citation evaluation benchmark**。它要求系统检索证据并生成带 citation 的答案，并从 fluency、correctness、citation quality 三个维度设计自动评估指标。论文还指出，即使最好的模型在 ELI5 上也经常缺少完整 citation support。([arXiv][2])

### 对你项目的价值

ALCE 适合用来支撑你报告里的 **citation quality evaluation**。

也就是说，它不应该被写成：

```text
我们参考 ALCE 来做生成。
```

而应该写成：

```text
我们参考 ALCE，把 citation quality 从 correctness 里拆出来，作为独立评估维度。
```

### 建议修改报告写法

建议改成：

```text
ALCE 的主要价值在于提醒我们：答案正确性和引用质量不是同一个问题。当前 VOC 系统中，一个 conclusion 即使语义上合理，也必须检查其 citation 是否完整、是否指向真正支撑该 conclusion 的证据。因此，本项目应新增 citation completeness、citation precision、citation contradiction 三类指标。
```

### 建议落地指标

建议新增：

```text
citation_completeness
    一个 claim 是否至少有一个有效 support_id

citation_precision
    被引用的 evidence 是否真的支持该 claim

citation_contradiction
    被引用证据中是否存在与 claim 相反的内容

claim_support_rate
    supported claims / all claims
```

## 3. Self-RAG

### 原文核心

Self-RAG 的核心是让模型在生成过程中自适应决定是否检索，并通过 reflection tokens 对检索内容和自身生成进行反思。它不是简单的 prompt trick，而是训练模型学习 retrieve、generate、critique。([arXiv][3])

### 对你项目的价值

Self-RAG 有价值，但**不适合你现在直接落地**。

原因是你的项目现在最紧急的问题是：

```text

```

而 Self-RAG 主要解决的是：

```text
什么时候检索
检索后如何自我反思
生成时如何动态控制
```

这更像下一阶段研究方向。

### 建议修改报告写法

原报告里如果把 Self-RAG 放在强相关，建议降级为：

```text
Self-RAG 适合作为 future work，用于设计 adaptive retrieval and critique mechanism。当前阶段不建议直接复现 Self-RAG，因为它涉及训练或至少模拟 reflection-token 风格的生成控制；本项目更适合先实现 lightweight critique fields，例如 retrieval_needed、evidence_sufficient、claim_supported。
```

### 可借鉴但不照搬的字段

```json

  "retrieval_needed": true,
  "evidence_sufficient": false,
  "critique": "The current evidence supports lens color but not long-duration running stability.",

}
```

## 8. CRAG

### 原文核心

CRAG 针对的问题是：RAG 很依赖检索文档相关性，如果检索错了，生成就会出问题。它提出 lightweight retrieval evaluator，根据检索质量触发不同 corrective action，并通过 decompose-then-recompose 过滤无关信息。([arXiv][8])

### 对你项目的价值

CRAG 很适合你现在的第二阶段。

你现在有：

```text

score_drift
top_k_count
answer_coverage
```

但这些还不能直接告诉系统下一步该干什么。CRAG 的价值是把指标变成动作：

```text
bad retrieval → rewrite query
modality miss → add image route
low recall → expand top-k
low precision → rerank/filter
```

### 建议修改报告写法

```text
CRAG 不只是“增加检索评估”，而是把 retrieval quality label 转换成 corrective action。因此当前项目应新增 RetrievalEvaluator，使 route_coverage、score_drift、answer_coverage 等字段能够驱动 rewrite_query、expand_topk、add_multimodal_route、filter_noise 等动作。
```

### 建议新增输出

```json
{
  "retrieval_quality_label": "mixed",
  "failure_reason": "modality_miss",
  "corrective_action": "add_multimodal_route",
  "explanation": "Text reviews support lens color, but image evidence is missing for tint darkness."
}
```

## 10. Query Rewriting for Retrieval-Augmented LLMs

### 原文核心

这篇论文提出 **Rewrite-Retrieve-Read**，重点不是改 reader，也不是改 retriever，而是改 search query，因为用户输入和检索所需知识之间存在 gap。它还提出用小模型作为 trainable rewriter，并用 reader feedback 训练。([arXiv][10])

### 对你项目的价值

这篇非常适合支撑你的 **intent-first question generation**。

你现在的问题生成会漂移，比如：

```text
太阳镜问题里出现鞋类词汇
普通用户问题被写成过度 optical engineering 问题
suggestion 过具体但证据不足
```

Query Rewriting 的启发是：

```text
不要直接生成自然语言问题，
先生成 retrieval intent，
再把 intent 渲染成检索 query。
```

### 建议修改报告写法

```text
Query Rewriting for RAG 的启发是：retrieval query 本身需要适配检索系统，而不能简单沿用自然语言问题。当前项目应将 question generation 拆成 retrieval intent generation 与 query rendering 两步，从而控制 aspect、desired modality、forbidden concepts、specificity bound。
```

### 建议数据结构

```json
{
  "aspect": "nose comfort",
  "user_concern": "discomfort around nose area during wearing",
  "desired_evidence_modality": "review",
  "forbidden_concepts": ["outsole", "midsole", "shoe lace"],
  "specificity_bound": "daily use, not technical optical measurement",
  "rendered_query": "nose area uncomfortable wearing sunglasses"
}
```

## 17. Position-Aware Tagging for ASTE

### 原文核心

ASTE 是抽取 target entity、sentiment、opinion span 三元组的任务。该论文认为三元组中的三个元素高度相关，因此提出 joint extraction，而不是 pipeline 式分阶段抽取。([arXiv][17])

### 对你项目的价值

它适合作为你设计：

```text
aspect-opinion-sentiment triplet
```

的理论依据。

但不建议直接复现它的模型，因为你现在主要是 LLM pipeline，不是训练序列标注模型。

### 建议修改报告写法

```text
Position-Aware Tagging 的主要启发是：aspect、opinion span、sentiment 不应被完全分离处理。当前项目可以在 LLM extraction schema 中显式要求三者共同输出，从而降低后续 relation 聚合的不稳定性。
```

### 建议把 review extraction 改成

```json
{
  "review_id": "review_0021",
  "opinion_units": [
    {
      "aspect": "lens tint",
      "opinion_span": "too dark indoors",
      "sentiment": "negative",
      "polarity_strength": "medium",
      "usage_context": "indoor or low-light environment"
    }
  ]
}
```

```text
NLP Architect 仅保留为历史参考，主要用于说明 domain-specific aspect/opinion lexicon 的工程价值。由于项目已归档，不建议将其作为当前系统实现的主要依赖或核心依据。
```

```text
After reviewing the related work, we revise the improvement plan as follows. RARR and ALCE should serve as the main methodological basis for claim-level evidence attribution and citation quality evaluation.
```
