# 高价值改进点调研与落地建议（核对稿）

本文档用于给当前项目做一轮可核对的研究与实施梳理。

目标不是重复介绍系统，而是聚焦三个问题：

1. 当前真实产物里最值得优先改的不足是什么
2. 每条不足对应哪些论文或开源实现值得参考
3. 哪些路线适合直接落地到当前仓库，哪些只适合作为中长期研究方向

本文档基于截至 2026-04-27 的仓库状态、真实运行结果与近期联网调研整理。

---

## 一、当前判断

当前系统已经完成了端到端多模态 VOC 分析主链，真实运行也已验证通过。但从真实产物来看，当前瓶颈已经不再是“链路能不能跑通”，而是以下三类问题：

- 证据能被召回，但不一定能稳定绑定到最终结论
- 检索质量有一些 proxy 指标，但还缺少对“答案为什么可信”的解释力
- 结构化 schema 已经比较完整，但真实 LLM 输出仍会在问题生成、建议生成和 evidence gap 表达上发生漂移

本轮优先级应从“继续补功能点”切换为“增强证据约束、评估解释力和生成稳定性”。

---

## 二、当前真实不足清单

以下不足不是抽象设想，而是已经在真实 artifact 中暴露出来的问题，尤其集中体现在：

- 02_outputs/4_reports/sunglasses/sunglasses_010_analysis.json

### 2.1 证据归因还不稳定

典型现象：

- lens tinting 会从 partial image support 退回成 review_only
- supporting_product_evidence 为空
- evidence_gaps 会退化成较泛的 “Missing ... substantiation”
- answer 与 suggestion 的支撑证据并不总能一一对应

这说明当前 schema 已经有 evidence 字段，但还缺少 claim 级别的 attribution 机制。

### 2.2 检索质量指标解释力不足

当前已有：

- route_coverage
- score_drift
- top_k_count
- answer_coverage 等 proxy 指标

但缺少：

- 每条答案到底由哪一路 route 真正支撑
- 检索结果是“不够多”还是“不够准”
- 哪些 claim 是 supported，哪些只是 partial supported

这导致 confidence_breakdown 难以真正填实，也限制了后续自动评测和误差定位。

### 2.3 问题生成与建议生成仍有漂移

虽然前面已经对 rubber insert、comfort、optical 相关问题加了多轮防漂移规则，但真实运行中仍然可能出现：

- 跨品类词汇
- 过技术化问题
- 过具体但证据不足的 suggestion 表达
- 真实 artifact 中 strengths / suggestions 语气回退得过于泛化

这说明当前主要依靠 prompt + 规则后处理，缺少更强的结构化约束。

### 2.4 aspect relation 只是初版

当前已能生成类似 negative_driver 的关系，但仍存在：

- 关系类型少
- relation summary 语句质量不稳定
- 关系仍主要依赖后验聚合规则，不是从更细粒度的 opinion structure 稳定推导

这意味着 aspect_relations 已经有价值，但还没有成为真正可复用的推理资产。

## 三、建议聚焦的四条改进主线

### 3.1 主线一：证据归因与引用链重构

这是当前最高优先级。

如果不先解决 attribution 问题，后面的评估、建议稳定化和论文表达都会受限，因为最终结果无法稳定回答“这句话是由哪些证据支持的”。

#### 代表性论文与开源

1. RARR
   - 论文：Researching and Revising What Language Models Say, Using Language Models
   - 链接：https://arxiv.org/abs/2210.08726
   - 价值：先找支撑，再对原回答做最小修订，适合 retrofit attribution

2. ALCE
   - 论文：Enabling Large Language Models to Generate Text with Citations
   - 链接：https://arxiv.org/abs/2305.14627
   - 价值：把 citation quality 作为独立维度评测，而不是只看 correctness

3. Self-RAG
   - 论文：Self-RAG: Learning to Retrieve, Generate, and Critique through Self-Reflection
   - 链接：https://arxiv.org/abs/2310.11511
   - 价值：把 critique 与 retrieval decision 合并到生成阶段，适合 future route

4. LlamaIndex CitationQueryEngine
   - 文档：https://developers.llamaindex.ai/python/examples/query_engine/citation_query_engine/
   - 价值：给出可直接参考的 citation chunking 和 source node 对齐工程模式

#### 与当前项目的相关性判断

- RARR：强相关，适合直接借思路
- ALCE：强相关，适合用来设计 citation 评测维度
- Self-RAG：中强相关，适合做下一阶段 adaptive retrieval / critique
- CitationQueryEngine：强相关，适合抄工程形态，不一定要引入 LlamaIndex 本体

#### 建议落地方式

建议先在当前 analysis artifact 中引入 claim 级支撑单元，而不是继续扩展大而全的 report 字段。

建议新增一层中间对象，例如：

- claim_text
- support_status：supported / partial / unsupported
- support_type：review / product_text / image / mixed
- support_ids：review_id、product evidence id、image region id
- route_sources：哪一路检索提供了该支撑

然后在 analysis_service 中增加一轮 post-hoc attribution：

- 先生成候选答案
- 再逐 claim 绑定证据
- 如果 claim 无法绑定，则降级表述或进入 evidence gap

#### 优先级

- 优先级：P0
- 类型：可立即工程落地

---

### 3.2 主线二：检索质量评估与 corrective retrieval

这是第二优先级。

当前最大问题不是完全没有指标，而是指标还不足以支撑调参与失败定位。现在的 proxy metrics 更像日志，不像评估结论。

#### 代表性论文与开源

1. Ragas
   - 开源：https://github.com/explodinggradients/ragas
   - 价值：提供 objective metrics、testset generation、feedback loop

2. TruLens
   - 开源：https://github.com/truera/trulens
   - 价值：RAG Triad、feedback function、组件级观察

3. DeepEval
   - 开源：https://github.com/confident-ai/deepeval
   - 价值：component-level eval，适合把 retriever / reranker / generator 拆开测

4. CRAG
   - 论文：Corrective Retrieval Augmented Generation
   - 链接：https://arxiv.org/abs/2401.15884
   - 价值：先做 retrieval evaluator，再决定 corrective action

5. Atlas
   - 论文：Atlas: Few-shot Learning with Retrieval Augmented Language Models
   - 链接：https://arxiv.org/abs/2208.03299
   - 价值：提醒索引内容、retriever 质量与任务表现强相关，更偏背景参考

#### 与当前项目的相关性判断

- Ragas：强相关，可借其 metric 设计方式
- TruLens：强相关，可借其 route 级 / component 级评估思路
- DeepEval：强相关，适合测试集与回归环节接入
- CRAG：强相关，适合指导 corrective retrieval 策略设计
- Atlas：中相关，更适合作为背景说明，不建议直接照搬

#### 建议落地方式

建议不要一次性接入完整第三方框架，而是先在本仓库中固化 4 个第一批指标：

1. claim support rate
2. context precision
3. route contribution
4. modality hit rate

在此基础上再新增一个 lightweight retrieval evaluator，对每个 question 输出：

- retrieval_quality_label：good / mixed / bad
- failure_reason：low_precision / low_recall / modality_miss / evidence_conflict
- corrective_action：rewrite_query / expand_topk / add_multimodal_route / keep_current

这一步完成后，route_coverage 和 score_drift 才能从“统计字段”升级为“决策输入”。

#### 优先级

- 优先级：P1
- 类型： attribution 之后立即推进

---

### 3.3 主线三：约束式问题生成与结构化输出校验

这是第三优先级，但它和第二条可以部分并行。

当前的问题不是完全不会问，而是偶发性地问偏、问过深、或者问成不适合当前商品语境的问题。类似问题也会传播到 suggestion 和 evidence gap 的表述层。

#### 代表性论文与开源

1. Query Rewriting for Retrieval-Augmented Large Language Models
   - 论文链接：https://arxiv.org/abs/2305.14283
   - 价值：把 Rewrite-Retrieve-Read 作为完整框架，强调 query 本身需要适配 retrieval

2. Outlines
   - 开源：https://github.com/dottxt-ai/outlines
   - 价值：在生成时强制满足 JSON schema、Pydantic、grammar 等约束

3. Guardrails
   - 开源：https://github.com/guardrails-ai/guardrails
   - 价值：通过 validator 做输入输出校验，适合挂 domain guard

4. Instructor
   - 开源：https://github.com/instructor-ai/instructor
   - 价值：基于 Pydantic 的轻量结构化输出约束，和当前仓库风格非常贴近

5. LangChain
   - 开源：https://github.com/langchain-ai/langchain
   - 价值：提供 orchestration 思路，但不建议把这个问题交给大框架本身解决

#### 与当前项目的相关性判断

- Query Rewriting：强相关，适合作为 retrieval 问题生成的上游改写逻辑
- Outlines：强相关，适合在 question schema 和 analysis schema 上做硬约束
- Guardrails：强相关，适合 domain-specific validator
- Instructor：强相关，而且更轻量，最适合先试
- LangChain：弱到中相关，主要作为编排启发

#### 建议落地方式

建议把当前 question generation 拆成两步：

1. 先生成 retrieval intent
   - aspect
   - user concern
   - desired evidence modality
   - forbidden concepts
   - specificity bound

2. 再把 retrieval intent 渲染成自然语言问题

并在输出后增加两层校验：

- schema validator：字段齐全且值域合法
- domain validator：不得出现跨品类词、过技术化 optical 词、与商品语境明显不符的材料或结构描述

如果不通过，则自动重写，而不是依赖后续人工修词。

#### 优先级

- 优先级：P1
- 类型：可与 retrieval evaluator 并行推进

---

### 3.4 主线四：aspect relation 与细粒度 opinion graph

这是第四优先级，更适合作为当前系统稳定之后的增强方向。

其价值主要体现在两点：

- 帮助报告生成从“列举 aspect”升级成“解释 aspect 如何相互影响”
- 帮助论文部分从普通 aspect summary 升级为更有结构感的方法表达

#### 代表性论文与开源

1. ABSA Survey
   - 论文：A Survey on Aspect-Based Sentiment Analysis: Tasks, Methods, and Challenges
   - 链接：https://arxiv.org/abs/2203.01054
   - 价值：对 compound ABSA 任务拆分很清楚，适合用来设计 relation taxonomy

2. PyABSA
   - 开源：https://github.com/yangheng95/PyABSA
   - 价值：覆盖 ATEPC、ASTE、ASQP、ACOS，多任务接口统一，工程参考价值高

3. Position-Aware Tagging for Aspect Sentiment Triplet Extraction
   - 论文链接：https://arxiv.org/abs/2010.02609
   - 价值：强调 aspect、opinion、sentiment triplet 联合抽取

4. NLP Architect
   - 开源：https://github.com/IntelLabs/nlp-architect
   - 价值：有 ABSA solution，但项目已归档，只保留启发意义

#### 与当前项目的相关性判断

- ABSA Survey：强相关，适合做任务建模参考
- PyABSA：强相关，适合参考其任务接口划分与输出结构
- Position-Aware Tagging：中强相关，适合方法设计参考
- NLP Architect：弱相关，只能作为历史参考

#### 建议落地方式

建议不要直接引入复杂图模型，而是分两步走：

1. 在 review_service 输出更接近 triplet / quadruple 的结构
2. 在 analysis_service 中把这些结构归并成 relation edges

建议第一批关系类型先收敛为：

- negative_driver
- positive_driver
- tradeoff
- co_occurs
- mitigated_by

这样可以先把 aspect_relations 从“附属字段”变成“稳定生成资产”，再决定是否继续升级到 graph-based reasoning。

#### 优先级

- 优先级：P2
- 类型：稳定化完成后的增强方向

## 四、哪些资料适合直接借，哪些只适合参考

### 4.1 适合直接借工程思路

- LlamaIndex CitationQueryEngine
- Ragas
- TruLens
- DeepEval
- Outlines
- Guardrails
- Instructor

这些资源的共同特点是：

- 工程接口清晰
- 问题定义和当前项目贴近
- 能直接转换成当前仓库中的 schema、service 或测试策略

### 4.2 适合借方法，不适合直接照搬

- RARR
- ALCE
- Self-RAG
- CRAG
- Query Rewriting for RAG

这些资源更适合作为“我们为什么这样设计”的方法依据，而不是直接引依赖。

### 4.3 只适合作为背景参考

- Atlas
- NLP Architect

它们能提供任务理解或系统背景，但对当前仓库的直接改造帮助有限。

## 五、已确认不适合当前问题的误抓项

本轮调研中已经识别出几项误抓资料，不建议继续引用到当前改进方案中：

- W04-3250：实际是机器翻译显著性检验，不是目标 ABSA 文献
- arXiv:1912.10206：是 GNN structural noise，与当前问题无关
- arXiv:2401.17050：是 ViTree 视觉分类可解释性，与当前 RAG / attribution 主线不匹配

这些内容应从后续正式引用清单中排除，避免论文与设计文档出现不准引用。

## 六、建议实施顺序

### 第一阶段：先修证据链

目标：让最终 report 中每个关键结论都能明确回答“由什么证据支持”。

建议动作：

1. 新增 claim-level attribution schema
2. 在 analysis_service 中增加 post-hoc attribution
3. 对 unsupported / partial supported claim 做表述降级
4. 为 answer / strengths / suggestions 统一补充 support status

### 第二阶段：再修评估解释力

目标：把当前 retrieval_quality 从日志字段升级为决策字段。

建议动作：

1. 引入 claim support rate 等第一批指标
2. 新增 retrieval evaluator
3. 补 corrective action 字段
4. 建立可复跑的 retrieval regression case

### 第三阶段：最后修生成稳定性

目标：减少 question 和 suggestion 漂移，抬高真实 artifact 一致性。

建议动作：

1. 把 question generation 改为 intent-first
2. 加 schema validator 和 domain validator
3. 用更轻量的 Instructor 或 Outlines 先做一版硬约束
4. 把 suggestion 生成也纳入同类约束链路

### 第四阶段：扩 relation graph

目标：让 aspect_relations 从“展示字段”变成“推理资产”。

建议动作：

1. review_service 输出更细的 opinion structure
2. analysis_service 聚合 relation edges
3. 扩 relation taxonomy
4. 再考虑更复杂的图结构表达

## 七、最终建议结论

如果只选一条最先做，应当优先做“证据归因与引用链重构”。

原因很直接：

- 它最能直接修复当前真实 artifact 中的支撑不稳问题
- 它会给 retrieval 评估和 question 稳定化提供更清晰的目标函数
- 它也最利于后续论文写作，因为方法与实验都需要稳定的可核验证据链

因此，建议当前总路线收敛为：

1. 先做 claim-level attribution
2. 再做 retrieval evaluator
3. 再做 schema-first question / suggestion generation
4. 最后扩展 aspect relation graph

这四步是递进关系，不建议颠倒。

---

## 八、供核对的简版结论

如果只保留一句话版本，可以概括为：

当前项目最值得优先投入的方向，不是继续堆新功能，而是把“结论和证据的绑定关系”做实；在此基础上，再把检索评估做成可解释、把问题生成做成强约束、把 aspect relation 做成稳定资产。
