# 项目开发、论文与成果综述

本文基于当前仓库截至 2026-04-13 的真实状态整理，目标是回答四个问题：

1. 我们是如何进行开发的
2. 论文当前处于什么状态
3. 当前整体成果是怎样的
4. 未来还要做哪些事情

这份文档不把“设想”和“已落地事实”混在一起，而是尽量以当前仓库中的代码、脚本、测试、输出目录和设计文档为依据进行总结。

---

## 一、先说结论：这个项目本质上是什么

这个项目不是单纯的评论情感分析，也不是普通的问答式 RAG，而是一个面向迪卡侬商品场景的、证据驱动的多模态 VOC 分析系统。

它要解决的问题不是“回答一个问题”，而是围绕一个商品产出一整套可追溯的分析结果，包括：

- 商品优点
- 商品缺点
- 用户关注点
- 使用场景印象
- 争议点或证据不足点
- 可回指到评论、商品文案和商品图片的改进建议

从当前仓库的设计判断来看，系统采用的是“分阶段保真”的路线，而不是把所有输入直接扔给一个大模型端到端生成。核心思想是：

- 商品侧尽量保留原始模态或结构化证据
- 评论侧尽快转成 aspect 级结构化对象
- 检索、聚合、生成都要受证据约束
- 全链路保留可解释性，便于审查、调试和误差分析

这也是为什么当前仓库会同时存在：

- 成体系的设计文档
- 可运行的 Python/FastAPI 工程
- 一键脚本和工作流
- 中间产物与最终报告
- 测试与验证样本

换句话说，这个仓库不是“只讲方法”的研究笔记，也不是“只写接口”的工程壳子，而是一个研究导向的工程化原型系统。

## 二、我们是如何进行开发的

### 2.1 开发方法不是先写代码，而是先把问题框定清楚

从当前文档结构可以看出，开发是从问题定义和系统分层开始的，而不是先写某个 API。

开发主线大致是：

1. 先明确项目定位，定义系统到底要解决什么问题。
2. 再建立总体架构，把系统拆成数据层、召回层、评论建模层、聚合层、生成与建议层、可解释性与评估层。
3. 然后针对每一层分别沉淀对象模型、设计原则、候选技术路线和取舍理由。
4. 最后才把这些设计转成可执行的工程模块和脚本。

这说明当前项目采用的是一种比较典型的“设计先行、工程跟进、验证反哺”的开发方式。

### 2.2 开发节奏采用的是 MVP 优先，而不是一次性做满

项目的实施路线非常明确：先跑通最小闭环，再逐步增强质量。

当前 MVP 的边界被控制得比较克制，重点是先把下面这条主链路打通：

1. 扫描原始商品数据
2. 生成标准化证据包
3. 对评论做 aspect 级结构化抽取
4. 基于 aspect 生成待澄清问题
5. 用问题驱动商品图文双路召回
6. 做聚合统计
7. 输出综合意见和改进建议
8. 把中间结果与最终报告全部落盘

与之对应，项目明确没有在第一阶段就把下面这些高复杂度能力一次性全部做完：

- 区域级图像证据与局部对齐
- 客户印象建模与更复杂的人群差异分析
- 多轮 agent 反思
- 更强的视觉 cross-encoder 级精排方案

这里要特别说明，这句话的含义不是“这些方向后来都已经完整落地”，而是“它们被放到分阶段推进路线中处理”。从当前仓库的真实状态看：

- reranker 精排已经进入主链路，但目前还是文本重排，不是最终形态的视觉 cross-encoder。
- 客户印象相关方向已经进入设计和方法表述，但还没有以完整客户画像聚类系统的形式进入主链路。
- 区域级证据、评论与图片局部对齐仍属于后续增强方向，当前主链路仍以对象级或整图级证据为主。
- 多轮 agent 反思在当前项目中也还不是已落地的核心流程。

这种做法的实际意义很大，因为它把实现风险拆小了。项目先验证“链路是否成立”，再验证“召回质量是否足够”，最后才推进更细粒度的研究版方案。

### 2.3 工程实现是围绕分层结构展开的

当前源码位于 05_src/decathlon_voc_analyzer，下层次已经和设计文档基本对齐：

- app：API 与配置入口
- schemas：结构化数据对象
- stage1_dataset：数据标准化相关能力
- stage2_review_modeling：评论建模能力
- stage3_retrieval：检索能力
- stage4_generation：生成能力
- workflows：工作流编排
- prompts：提示词管理

这说明开发不是“按接口文件堆起来”的，而是尽量让目录结构反映系统方法本身。

### 2.4 实际开发采用了“文档、代码、脚本、产物”并行推进

从仓库内容看，开发不是只靠源码推进，而是四条线并行：

#### 第一条线：设计文档线

设计文档负责固定问题定义、架构分层、对象关系、评估方式和技术选型，避免开发过程中路线漂移。

#### 第二条线：源码实现线

源码把设计中的各阶段能力转成真实模块，并通过 FastAPI 暴露成接口。

当前已经能看到的接口包括：

- /health
- /api/v1/dataset/overview
- /api/v1/dataset/normalize
- /api/v1/index/overview
- /api/v1/index/build
- /api/v1/reviews/extract
- /api/v1/analysis/product

#### 第三条线：脚本与工作流线

项目提供了一键工作流脚本，可以直接把标准化、索引、分析串起来运行，而不是要求每一步都手工调用。

这说明开发过程中已经考虑了“如何稳定复现一次完整实验”。

#### 第四条线：落盘产物线

当前 02_outputs 目录不是临时缓存，而是整个分析流程的产出区，里面明确区分了：

- 标准化结果
- aspect 中间结果
- 索引与向量库目录
- 最终分析报告
- 中文审查产物与人工审核材料

这种组织方式很适合研究和工程并行推进，因为它让每一阶段的结果都可审查、可复现、可比较。

### 2.5 开发过程本质上是“问题驱动检索”替代“评论直接检索”

从推进记录可以看到，当前实现中一个非常关键的路线升级是：

- 早期思路更接近“拿评论直接做召回”
- 后来稳定切换为“评论先抽 aspect，再生成问题，由问题驱动召回”

这一步很关键，因为它把检索从“直接匹配原句”提升成“围绕可验证问题寻找证据”。

好处包括：

- 能更明确地区分召回错、问题错、抽取错
- 更容易做误差分析
- 更利于后续把检索和生成解耦
- 更符合论文中的方法设计表达

换句话说，当前项目真正的检索锚点不是评论原文，而是由评论抽象出来的“待澄清问题”。

### 2.6 开发过程不是拍脑袋试，而是始终带测试和验证样本

当前仓库里已经存在完整测试集，覆盖范围包括：

- 数据集服务与 API
- 评论抽取服务与 API
- 问题生成服务
- 索引后端与索引服务
- embedding 与 reranker
- 分析服务与分析 API
- prompt 注册

我在当前环境中实际运行了测试，结果为：

- 27 项测试通过

这说明项目已经形成稳定基线，当前成果不是只在文档层面成立。

除此之外，项目还保留了真实样本验证记录，尤其是 backpack_010 这类单商品完整跑通样本，用来证明：

- 标准化能跑通
- aspect 抽取能跑通
- 问题生成能跑通
- 图文检索能跑通
- 最终分析和建议也能产出

因此，当前开发方式可以概括为一句话：

先用文档把方法论和边界收紧，再用 MVP 把链路跑通，然后用测试、样本和落盘产物持续固化成果。

## 四、当前整体成果是怎样的

整体成果可以分成四类：文档成果、工程成果、数据与报告成果、研究方法成果。

### 4.1 文档成果：已经形成一套可阅读、可答辩、可扩展的文档系统

当前 03_docs 目录并不是零散说明，而是按主题分层组织的知识系统，覆盖：

- 项目定位与阅读导览
- 总体架构与技术选型
- 数据层与对象关系
- 召回层演进
- 评论建模
- 生成与建议
- 可解释性与评估
- 实施路线与推进状态
- 参考资料与引用目录
- 审查与覆盖检查

这套文档系统的价值在于，它已经把“原始对话中的想法”转成“可沉淀、可复用、可继续写作”的稳定资产。

### 4.2 工程成果：已经形成可运行的 MVP 闭环系统

从代码和脚本状态看，当前工程成果至少包括：

- 基于 Python 3.11+ 的项目结构和依赖管理
- FastAPI 应用入口与 API 路由
- 数据集扫描和标准化能力
- 评论预处理、低信息评论过滤和 aspect 抽取
- 问题生成服务
- 索引构建与检索后端抽象
- 本地索引后端与 Qdrant 后端骨架/可用路径
- embedding 与 reranker 接入
- 综合分析与改进建议生成
- 一键单商品工作流脚本

更关键的是，这些成果不是孤立模块，而是已经能串成一条完整主链路。

### 4.3 数据与报告成果：已经有真实样本的中间结果和最终结果

当前 02_outputs 目录已经明确沉淀了完整的过程产物：

- 标准化证据包
- aspect 级评论建模结果
- 本地索引与 Qdrant 目录
- 最终分析报告
- 中文审查报告
- 面向人工审核的 Markdown 文件

这类成果非常重要，因为它说明项目输出的不只是代码执行结果，而是：

- 可检查的中间对象
- 可比较的实验产物
- 可给人审阅的最终报告

以 backpack_010 为例，当前已经能看到一份完整分析结果，其中包含：

- 评论预处理结果
- aspect 抽取结果
- 待澄清问题
- 召回证据
- 最终 strengths / weaknesses / controversies / suggestions

这说明系统已经能够对真实商品跑出一份带证据链的分析档案。

### 4.4 研究方法成果：项目已经形成了清晰的方法论主张

从现有材料看，当前项目最核心的方法成果不只是“做成了一个系统”，而是形成了一套比较明确的研究主张：

1. 商品分析不能只依赖评论文本，需要把商品图像和商品文案一起纳入证据体系。
2. 评论不应直接驱动生成，而应先转成 aspect 级结构化对象。
3. 检索不应直接拿评论原句做 query，而应由 aspect 生成可核验问题，再由问题驱动证据召回。
4. 生成结果不应只输出最终结论，而应保留完整证据链和中间可解释对象。
5. 评估不应只看最终报告好不好，还应分层评估抽取、检索、聚合和生成。

如果把这些主张放进论文语境，它们已经具备“方法章节核心贡献点”的雏形。

### 4.5 当前成果的阶段判断：不是最终生产系统，但已经是可靠的研究原型

结合当前实现边界，更准确的判断应该是：

- 它已经不是概念稿，也不是纯脚手架。
- 它已经具备真实数据、真实流程、真实输出和测试基线。
- 它已经可以支撑方法验证、案例分析和后续论文写作。
- 但它还不是最终研究版，也不是生产级最终形态。

当前仍然存在的边界包括：

- 图像检索仍有代理文本成分，尚未全面进入更强的原始视觉表示阶段。
- reranker 已接入，但还没有推进到更细粒度的视觉 cross-encoder 方案。
- 区域级证据、局部对齐、客户印象建模等高级能力还未全部进入主链路。

因此，当前最准确的成果定位是：

一个已跑通、可验证、可扩展、可写入论文的方法型工程原型系统。

## 五、未来还要做哪些事情

未来工作主要不是继续堆新接口，而是把当前已经识别出的关键短板逐步补齐。结合当前实现状态和《VOC分析系统关键设计问题答复》中提到但尚未解决的部分，后续建议重点推进以下几类工作。

### 5.1 置信度体系升级

当前置信度更多还是经验值和模型自报，还不是校准后的、可解释的客观概率。后续应把置信度从单一分数拆成至少四层：

- 评论抽取置信度
- 问题质量置信度
- 证据支撑置信度
- 最终结论置信度

同时把跨模态一致性、证据覆盖度、冲突证据和 rerank 稳定性纳入统一计算框架。

### 5.2 结构化过程项与建议生成升级

当前系统已经能直接输出 strengths、weaknesses 和 suggestions，但还没有把“过程类 item”正式建成主链路对象。后续应补上：

- 评论观察项
- 证据支撑项
- 证据缺失项
- 归因判断项
- owner 分类项

建议生成也需要从“直接给建议”升级为“先形成结构化推理轨迹，再汇总成最终建议”。这样更利于验收、去重、训练和误差分析。

### 5.3 分层评估与反馈闭环

当前项目已有测试和样本验证，但还没有形成完整的数据闭环。后续需要正式建立：

- 评论抽取层评估
- 问题生成层评估
- 检索层评估
- 结论层评估
- 建议层评估

并把人工标注、错误聚类、回归样本和重跑对比串成稳定反馈循环，而不是仅依赖单次人工阅读结果。

### 5.4 检索策略进一步升级

当前系统已经完成问题驱动召回、双路检索和 reranker 接入，但还有几项关键能力没有进入主链路：

- 对优点做轻召回、对缺点/争议/建议做强召回的非对称策略
- 更强的多模态 embedding
- 更强的 cross-modal reranker
- 区域级图像证据定位
- 评论与图片的局部对齐

这部分会直接决定后续系统从“可跑通”走向“召回质量稳定”。

### 5.5 去重、归因与结果组织

当前结果里仍然可能出现语义近重复、证据池复用过宽、页面问题与产品问题混淆等情况。后续需要补齐：

- item 级语义去重
- evidence overlap 去重
- weakness / controversy / suggestion 的 owner 分类
- 产品问题与页面表达问题的明确拆分
- 产品改进建议与页面表达优化建议的分栏输出

这部分对于结果可信度和业务可用性都非常关键。


### 5.6 结果展示与人工审核界面

当前输出已经有 JSON 结果和部分人工审核材料，但面向展示和审阅的交互层还没有正式完成。后续可以继续推进：

- HTML 结果页
- 可锚点跳转的分区展示
- 证据展开与回指
- 商品优点、缺点、争议、建议的分栏阅读体验

这部分对于项目验收、答辩展示和人工校对都很有帮助。
