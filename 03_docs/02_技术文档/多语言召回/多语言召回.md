

# 基于语言感知与证据保留的多语言 RAG 方法论

## 摘要

在多语言商品评论分析任务中，商品信息与用户评论往往分布在不同语言中。例如，商品标题、规格说明和官方描述可能是韩文，而用户评论可能是英文。在这种情况下，系统不仅要能够跨语言召回相关证据，还要避免某一种语言在检索排序中占据不合理优势。

传统做法通常是将所有文档预先翻译成一种统一语言，例如英文，然后再建立索引。然而，这种方法存在明显问题：第一，翻译可能丢失原文中的细粒度语义；第二，大规模评论翻译会增加系统成本；第三，翻译后的文本会削弱原文证据的可追溯性。

因此，本文提出一种 **语言感知且证据保留的多语言 RAG 方法**，即 **Language-Aware Evidence-Preserving Multilingual RAG，简称 LAEP-mRAG**。该方法不把所有内容提前翻译成同一种语言，而是保留原文，使用多语言嵌入模型进行初步召回，再通过语言平衡候选生成、查询翻译增强、统一语言重排和证据保留式生成来提升跨语言检索和回答质量。

该方法的核心思想是：
**召回阶段保留多语言原文，排序阶段显式处理语言偏置，生成阶段使用归一化证据辅助推理，同时保留原文证据用于引用和解释。**

---

# 1. 引言

在真实商品评论分析系统中，数据语言往往是不统一的。例如，一个韩国电商平台上的商品页面可能使用韩文描述：

```text
어깨끈은 장시간 착용 시 압박을 줄이도록 설계되었습니다.
```

意思是：

```text
肩带被设计为在长时间佩戴时减少压力。
```

但用户评论可能是英文：

```text
After 2 hours, the straps dig into my shoulders.
```

意思是：

```text
两个小时后，肩带会压进我的肩膀。
```

当用户用中文提问：

```text
这款背包长时间背会不会压肩？
```

系统必须同时理解：

* 韩文官方商品描述；
* 英文真实用户评论；
* 中文用户问题；
* 最终还要用中文生成答案。

这就是典型的 **多语言 RAG（Multilingual Retrieval-Augmented Generation, mRAG）** 问题。

早期或简单工程方案可能会选择：
**先把所有韩文商品信息和英文评论全部翻译成英文，然后只建立英文索引。**

但是，这种方法并不理想。因为商品评论中的表达往往很细，例如：

```text
dig into my shoulders
```

并不只是普通的“不舒服”，而是带有“压入肩膀”“勒肩”“持续压力”的含义。如果翻译不准确，后续分析就会丢失关键证据。

近年的多语言 RAG 研究也表明，多语言 RAG 的问题不仅是“能不能召回”，还包括：是否会产生语言偏置、是否会错误排序、是否会出现混合语言生成、是否会在目标语言中回答不稳定等问题 [1]。

因此，本文将多语言商品评论分析建模为一个 **语言感知的多阶段检索与生成问题**，而不是简单的“多语言 embedding 相似度排序”问题。

# 2. 相关工作

## 2.1 多语言 RAG 的基本问题

多语言 RAG 指的是查询、文档和答案可能处于不同语言中的检索增强生成任务。已有研究指出，即使使用多语言检索器和多语言生成模型，系统仍然容易出现一些问题，例如：

* 检索结果和问题语言不一致；
* 生成答案时出现中英韩混杂；
* 模型误读异语言证据；
* 检索到了相关文本，但最终没有正确使用；
* 回答语言偏离用户要求。

Chirkova 等人的研究表明，mRAG 在多语言上下文中容易出现 code-switching、流畅性下降、错误读取检索文档和无关检索等问题 [1]。这说明多语言 RAG 的难点不仅在检索阶段，也存在于生成阶段。

---

## 2.2 多语言嵌入模型与共享语义空间

多语言嵌入模型的作用是把不同语言的文本映射到同一个向量空间中。这样一来，韩文的“편안하다”、英文的“comfortable”和中文的“舒适”可以在向量空间中靠得比较近。

例如：

| 文本          | 语言 | 语义 |
| ----------- | -- | -- |
| 편안하다        | 韩文 | 舒适 |
| comfortable | 英文 | 舒适 |
| 舒适          | 中文 | 舒适 |

这类模型可以支持跨语言召回。比如用户用中文问“肩带舒不舒服”，系统仍然可能召回韩文商品说明和英文用户评论。

BGE-M3 是代表性多语言嵌入模型之一。它支持 100 多种语言，并且同时支持 dense retrieval、sparse retrieval 和 multi-vector retrieval，因此适合多语言、多粒度、多功能检索任务 [2]。

但是，仅仅把不同语言放进同一个语义空间是不够的。因为“能召回”不等于“能正确排序”。

---

## 2.3 混合检索

在真实系统中，只使用向量检索通常不够稳定。原因是：

* 向量检索擅长语义相似；
* 关键词检索擅长精确匹配；
* 商品系统里有大量品牌名、型号、尺寸、颜色、规格参数；
* 这些信息不能完全依赖语义相似度。

例如：

```text
Quechua NH Arpenaz 100 10L
```

这种商品型号最好通过关键词检索匹配，而不是只靠向量语义。

因此，现代检索系统通常使用 **混合检索（Hybrid Retrieval）**，即同时使用：

* dense retrieval；
* sparse retrieval；
* BM25；
* RRF 融合；
* reranker 重排。

Azure AI Search 的工程文档也说明，hybrid search 可以将 vector search 和 full-text search 结合，并通过 RRF 融合结果 [3]。

这说明，在多语言商品评论分析中，比较稳妥的方式不是单独使用多语言 embedding，而是使用：

```text
多语言向量检索 + 关键词检索 + 结果融合 + 重排
```

---

## 2.4 多语言检索中的语言偏置

多语言 RAG 中一个非常关键的问题是 **语言偏置（Language Bias）**。

也就是说，即使系统使用的是多语言 embedding，模型仍然可能偏向某一种语言。例如：

* 偏向英文；
* 偏向高资源语言；
* 偏向 query 所在语言；
* 偏向训练数据中占比更高的语言；
* 偏向与 query 同语言的文档。

Park 和 Lee 的研究指出，多语言 RAG 系统中的 retriever 可能偏好高资源语言或查询语言，但这种偏好并不一定能提高最终回答质量 [4]。

这对商品分析系统非常重要。比如用户问：

```text
这款背包长时间背会不会压肩？
```

系统里有两条证据：

```text
韩文官方说明：
肩带设计为减少长时间佩戴压力。
```

```text
英文用户评论：
After 2 hours, the straps dig into my shoulders.
```

如果系统偏向韩文，它可能过度相信官方说明；
如果系统偏向英文，它可能忽略官方规格描述。
两种情况都会导致分析不完整。

因此，最新方法不再只是依赖一个全局 top-k 排序，而是显式引入语言控制机制。

---

## 2.5 跨语言排序困难

在多语言 RAG 中，真正困难的地方往往不是“有没有相关文档”，而是“不同语言的相关文档能不能被正确排到前面”。

Amiraz 等人的研究发现，在阿拉伯语—英语企业语料场景中，RAG 的主要瓶颈来自跨语言排序问题。也就是说，相关证据可能已经存在于语料库中，但当 query 和 evidence 语言不一致时，系统很难把它排到前面 [5]。

这说明，对于多语言商品评论分析来说，只做：

```text
query → multilingual embedding → global top-k
```

是不够的。

更合理的做法是：

```text
query → 多语言增强 → 分语言召回 → 候选合并 → 统一语言重排
```

---

## 2.6 查询增强与去偏

DELTA 方法进一步指出，过去很多所谓“模型偏向英文”的现象，可能并不完全是模型天然偏向英文，而是受到数据集证据分布和 query-document 语言对齐关系的影响。该方法强调，retriever 往往更偏好 query 和 document 同语言的情况，因此可以利用语言偏好信号做 query-level augmentation，从而减少语言偏置 [6]。

这对本文方法有直接启发：

系统不应该只用用户原始 query 搜一次，而应该生成多个语言版本的 query，例如：

```text
中文 query：
这款背包长时间背会不会压肩？

英文 query：
Are the shoulder straps comfortable for long-term carrying?

韩文 query：
이 백팩은 오래 착용했을 때 어깨끈이 편안한가요?
```

然后分别进入不同语言候选池进行召回。

---

## 2.7 后置语言归一化

另一个关键问题是：
翻译到底应该放在什么时候？

简单系统会在检索前就把所有文档翻译成英文。但是 CrossRAG 等方法表明，更合理的方式不是完全取消翻译，而是把翻译放到检索之后，用于证据归一化和生成稳定化 [7]。

也就是说：

```text
不推荐：
所有文档 → 先翻译 → 建库 → 检索

推荐：
原文建库 → 多语言召回 → 证据归一化 → 重排 / 生成
```

这样既能利用多语言检索扩大证据范围，又能在生成阶段减少混合语言上下文带来的不稳定性。

# 3. 方法总体设计

本文提出的 **LAEP-mRAG** 方法包含八个核心步骤：

```text
原文多语言文档
        ↓
原文索引与元数据保存
        ↓
多语言 dense + sparse 混合检索
        ↓
查询翻译与查询增强
        ↓
语言平衡候选生成
        ↓
证据语言归一化
        ↓
多语言 / 统一语言重排
        ↓
证据保留式生成与 claim-level attribution
```

它的核心目标有四个：

1. **提高跨语言召回率**
   让韩文商品信息和英文评论都能被召回。

2. **减少语言偏置**
   避免英文、韩文或 query 语言垄断 top-k。

3. **提高生成稳定性**
   避免 LLM 在混合语言上下文中误读、乱翻译或语言漂移。

4. **保留证据可解释性**
   最终答案中的每个关键结论都能追溯到原文证据。

---

# 4. 具体方法

## 4.1 原文索引与元数据保存

第一步不是翻译所有文档，而是保留所有文档的原文。

每条文本 chunk 保存如下字段：

```json
{
  "chunk_id": "review_0021",
  "product_id": "backpack_001",
  "text_original": "After 2 hours, the straps dig into my shoulders.",
  "language": "en",
  "doc_type": "review",
  "aspect": "strap_comfort",
  "rating": 2,
  "text_normalized": null,
  "embedding": "[...]"
}
```

韩文商品说明可以保存为：

```json
{
  "chunk_id": "spec_0003",
  "product_id": "backpack_001",
  "text_original": "어깨끈은 장시간 착용 시 압박을 줄이도록 설계되었습니다.",
  "language": "ko",
  "doc_type": "product_spec",
  "aspect": "strap_comfort",
  "rating": null,
  "text_normalized": null,
  "embedding": "[...]"
}
```

这里最重要的是保存：

| 字段                | 作用               |
| ----------------- | ---------------- |
| `text_original`   | 保留原文证据           |
| `language`        | 用于语言平衡召回         |
| `doc_type`        | 区分官方说明、规格、评论     |
| `aspect`          | 标注商品方面，例如舒适性、耐用性 |
| `text_normalized` | 后续用于统一语言重排和生成    |
| `embedding`       | 用于多语言向量检索        |

这样设计的好处是：
系统既可以使用多语言 embedding 做跨语言召回，也可以在最终报告中展示原始证据，而不是只展示翻译文本。

---

## 4.2 多语言混合检索

检索阶段同时执行两种检索：

```text
Dense Retrieval:
使用多语言 embedding 进行语义召回

Sparse Retrieval / BM25:
使用关键词检索进行精确匹配
```

例如用户问：

```text
这款背包长时间背会不会压肩？
```

dense retrieval 负责找到语义相关内容：

```text
After 2 hours, the straps dig into my shoulders.
```

BM25 或 sparse retrieval 负责匹配精确词：

```text
strap
shoulder
어깨끈
backpack
model number
```

然后使用 RRF 合并结果：

```text
candidate_pool = RRF(dense_results, sparse_results)
```

这种设计比单独使用 embedding 更稳定，因为商品分析中既有语义问题，也有大量精确属性和型号匹配问题 [3]。

---

## 4.3 查询翻译与查询增强

为了降低 query 和 document 语言不一致造成的召回损失，系统会生成多个 query 版本。

例如原始 query 是中文：

```text
这款背包长时间背会不会压肩？
```

系统生成：

```text
English query:
Are the shoulder straps comfortable for long-term carrying?

Korean query:
이 백팩은 오래 착용했을 때 어깨끈이 편안한가요?
```

然后进行多路检索：

```text
q_zh → 全局多语言索引
q_en → 英文评论索引
q_ko → 韩文商品说明索引
```

这里的翻译不是为了替代多语言检索，而是为了增强 query-document 的语言对齐。DELTA 等研究也说明，语言偏好和 query-document 对齐关系会影响多语言 RAG 的检索表现，因此 query-level augmentation 是重要的去偏手段 [6]。

---

## 4.4 语言平衡候选生成

普通 RAG 通常是：

```text
query → 全库检索 → top-k
```

但是在多语言系统中，这样会有问题。因为 top-k 可能全部来自一种语言。

例如：

```text
Top-10 全部是英文评论
```

或者：

```text
Top-10 全部是韩文官方说明
```

这都会导致证据不完整。

因此，LAEP-mRAG 使用语言平衡候选生成：

```text
C_ko = retrieve(q_ko, language="ko", k=K_ko)

C_en = retrieve(q_en, language="en", k=K_en)

C_global = retrieve(q_original, language="any", k=K_global)

C = merge(C_ko, C_en, C_global)
```

也就是说，系统先保证不同语言的证据都有机会进入候选池，然后再由 reranker 判断谁更重要。

这里要特别注意：
**语言平衡不是最终答案里强行平均使用每种语言的证据。**

它只是保证在候选生成阶段：

```text
韩文证据有机会进入；
英文证据有机会进入；
全局高分证据也有机会进入。
```

最后是否采用，仍然取决于重排模型和证据质量。

这种做法可以缓解跨语言排序困难。相关研究也表明，balanced retrieval 和 query translation 对跨语言 RAG 表现有明显帮助 [5]。

---

## 4.5 证据语言归一化

候选证据被召回后，系统会为每条证据准备两个版本：

```text
原文证据：
text_original

归一化证据：
text_normalized
```

例如：

```text
原文韩文：
어깨끈은 장시간 착용 시 압박을 줄이도록 설계되었습니다.

归一化英文：
The shoulder straps are designed to reduce pressure during long-term use.
```

再例如：

```text
原文英文：
After 2 hours, the straps dig into my shoulders.

归一化英文：
After two hours, the shoulder straps become uncomfortable and press into the shoulders.
```

这样做的原因是：

* 原文用于证据追踪；
* 归一化文本用于重排和生成；
* 重排模型更容易在统一语言下比较语义；
* 生成模型更容易基于统一语言证据进行总结。

这与 CrossRAG 的思想一致：不一定要在检索前翻译全部语料，而是在检索后对 retrieved evidence 做语言归一化，从而提高生成阶段的稳定性 [7]。

---

## 4.6 多语言或统一语言重排

初步召回之后，系统需要使用 reranker 进行精排。

重排有两种方式。

### 方式一：直接使用多语言 reranker

例如使用 Jina reranker v3 或 Cohere rerank 这类多语言重排模型。Jina reranker v3 被设计为 multilingual document reranker，Cohere 的 rerank 模型也支持 100 多种语言 [8][9]。

此时输入可以是：

```text
query: 这款背包长时间背会不会压肩？

document 1: 어깨끈은 장시간 착용 시 압박을 줄이도록 설계되었습니다.

document 2: After 2 hours, the straps dig into my shoulders.
```

模型直接判断哪个 evidence 更相关。

### 方式二：统一语言后再 rerank

另一种更稳的做法是：

```text
query → 翻译 / 归一化为英文
candidate evidence → 翻译 / 归一化为英文
reranker → 在统一语言下排序
```

例如：

```text
Normalized query:
Are the shoulder straps comfortable for long-term carrying?

Normalized evidence 1:
The shoulder straps are designed to reduce pressure during long-term use.

Normalized evidence 2:
After two hours, the shoulder straps become uncomfortable and press into the shoulders.
```

这种方式可以减少语言差异对 reranker 的影响。

最终排序分数可以设计为：

```text
final_score =
  α · dense_score
+ β · sparse_score
+ γ · reranker_score
+ δ · evidence_type_score
- η · redundancy_penalty
```

其中：

| 分数                    | 含义        |
| --------------------- | --------- |
| `dense_score`         | 向量语义相似度   |
| `sparse_score`        | 关键词匹配分数   |
| `reranker_score`      | 重排模型相关性分数 |
| `evidence_type_score` | 证据类型权重    |
| `redundancy_penalty`  | 重复证据惩罚    |

---

## 4.7 证据类型感知排序

商品评论分析不能只看“文本相似度”。因为不同证据类型的价值不同。

例如：

| 证据类型   | 作用       |
| ------ | -------- |
| 官方商品描述 | 说明设计目标   |
| 商品规格   | 提供客观参数   |
| 正面评论   | 说明用户满意点  |
| 负面评论   | 暴露真实问题   |
| 图片证据   | 支持视觉判断   |
| 聚合评论方面 | 说明问题是否普遍 |

对于问题：

```text
这款背包长时间背会不会压肩？
```

下面这条用户评论：

```text
After 2 hours, the straps dig into my shoulders.
```

可能比官方说明：

```text
The shoulder straps are designed for comfort.
```

更重要。

因为用户问的是“真实使用体验”，不是“官方设计目标”。

所以系统需要区分：

```text
官方声称：
肩带被设计为舒适。

用户体验：
两小时后肩带压肩。

综合判断：
官方设计目标与真实使用反馈存在一定冲突。
```

这种 evidence-type-aware ranking 对商品评论分析尤其重要。

---

## 4.8 证据保留式生成

生成阶段不能只把混合语言文本全部丢给 LLM。更合理的方式是给模型两层证据：

```text
第一层：归一化证据，用于推理和总结

第二层：原文证据，用于引用和可追溯性
```

例如：

```text
[Normalized Evidence]
E1: The shoulder straps are designed to reduce pressure during long-term use.
E2: A user reported that after two hours, the straps pressed into the shoulders.

[Original Evidence]
E1_original: 어깨끈은 장시간 착용 시 압박을 줄이도록 설계되었습니다.
E2_original: After 2 hours, the straps dig into my shoulders.
```

生成模型输出：

```text
官方说明称该背包肩带经过减压设计，因此从设计目标上看，它强调长时间佩戴舒适性。但是，英文用户评论显示，实际使用超过两小时后可能出现压肩问题。因此，该产品在短时间佩戴时可能较舒适，但长时间负重场景下仍存在肩部压力风险。
```

这种生成方式有三个好处：

1. **回答更稳定**
   因为模型主要基于归一化证据推理。

2. **证据可追溯**
   因为每条结论仍然能追溯到原文。

3. **减少语言混杂**
   因为最终回答可以被明确限制为中文。

已有 mRAG 研究表明，混合语言上下文可能导致 code-switching 和生成不稳定，因此生成阶段需要显式控制目标语言和证据使用方式 [1]。

---

# 5. 算法流程

```text
Algorithm 1: LAEP-mRAG

Input:
    用户查询 q
    多语言文档集合 D
    目标回答语言 L_target
    商品 ID p

Output:
    基于证据的回答 A
    claim-evidence attribution map M

Step 1: 查询理解
    检测查询语言 L_q
    抽取商品方面 aspect
    生成多语言查询集合：
        Q = {q_original, q_en, q_ko, ...}

Step 2: 多语言混合召回
    对每个 query variant q_i：
        执行 dense retrieval
        执行 sparse retrieval / BM25
        使用 RRF 融合结果

Step 3: 语言平衡候选生成
    分别召回：
        C_ko = 韩文候选
        C_en = 英文候选
        C_global = 全局候选
    合并：
        C = merge(C_ko, C_en, C_global)

Step 4: 证据归一化
    对每个候选证据 c：
        保存 c.text_original
        生成或读取 c.text_normalized

Step 5: 重排
    使用 normalized query 和 normalized evidence 计算相关性
    加入 evidence type score
    去除低质量和重复证据

Step 6: 证据选择
    选出最终证据集合 E
    保存：
        原文
        归一化文本
        语言
        文档类型
        aspect
        retrieval score
        reranker score

Step 7: 答案生成
    使用归一化证据进行推理
    使用原文证据进行引用
    按目标语言 L_target 生成答案

Step 8: Claim-level Attribution
    对每个生成结论：
        找到支持证据
        标注为：
            supported
            partially supported
            conflicting
            unsupported

Return:
    Answer A
    Attribution Map M
```

# 6. 案例说明

## 6.1 输入

```text
商品信息语言：
韩文

评论语言：
英文

用户问题：
这款背包长时间背会不会压肩？
```

---

## 6.2 召回证据

韩文官方说明：

```text
어깨끈은 장시간 착용 시 압박을 줄이도록 설계되었습니다.
```

英文用户评论：

```text
After 2 hours, the straps dig into my shoulders.
```

---

## 6.3 系统分析

韩文官方说明支持：

```text
该商品的肩带设计目标是减少长时间佩戴时的压力。
```

英文用户评论支持：

```text
实际使用超过两小时后，用户可能会感到肩带压肩。
```

因此系统不能简单回答：

```text
这款背包很舒服。
```

更准确的回答应该是：

```text
官方设计目标是舒适和减压，但真实用户反馈显示，长时间使用时仍可能出现压肩问题。
```

---

## 6.4 最终回答示例

```text
该背包在官方设计上强调肩带减压，因此短时间或中等强度使用时可能具有一定舒适性。但是，用户评论显示，在连续背负约两小时后，肩带可能会压入肩部，说明它在长时间负重场景下仍存在压肩风险。综合来看，该产品不能简单判断为“长时间背负舒适”，更准确的结论是：设计目标偏向舒适，但实际长时间使用体验存在负面反馈。
```

---

## 6.5 Claim-Level Attribution 示例

| 生成结论          | 支持证据        | 证据类型  | 语言   | 状态        |
| ------------- | ----------- | ----- | ---- | --------- |
| 肩带设计目标是减压     | 韩文商品说明      | 官方规格  | 韩文   | Supported |
| 长时间背负后可能压肩    | 英文用户评论      | 用户评论  | 英文   | Supported |
| 官方描述与用户体验存在冲突 | 韩文说明 + 英文评论 | 跨证据比较 | 混合语言 | Supported |

# 7. 实验设计

## 7.1 检索评估

检索模块可以使用以下指标：

| 指标                     | 含义              |
| ---------------------- | --------------- |
| Recall@k               | 相关证据是否进入 top-k  |
| nDCG@k                 | 相关证据是否排在前面      |
| MRR                    | 第一个相关证据出现得有多早   |
| Language Coverage Rate | 不同语言证据是否都被覆盖    |
| Evidence Type Coverage | 官方信息和用户评论是否都被覆盖 |

其中，**Language Coverage Rate** 对多语言 RAG 特别重要。

因为一个系统可能 Recall@k 很高，但如果召回结果全部来自英文评论，而没有韩文官方说明，那么它仍然是不完整的。

---

## 7.2 语言偏置评估

可以构造四种 query-document 组合：

| Query 语言 | 支持证据语言 | 示例          |
| -------- | ------ | ----------- |
| 中文       | 韩文     | 中文用户问韩文商品规格 |
| 中文       | 英文     | 中文用户问英文评论内容 |
| 英文       | 韩文     | 英文用户问韩文商品规格 |
| 韩文       | 英文     | 韩文用户问英文评论内容 |

然后比较以下方法：

```text
1. global top-k retrieval
2. language-balanced retrieval
3. query-translation retrieval
4. full LAEP-mRAG
```

主要观察：

```text
当 query 和 evidence 语言不一致时，系统性能是否明显下降？
```

已有研究显示，跨语言 ranking failure 是 domain-specific bilingual RAG 的主要瓶颈之一 [5]。因此，这个实验可以直接证明语言平衡召回和查询增强是否有效。

---

## 7.3 生成质量评估

生成阶段可以评估：

| 指标                          | 含义              |
| --------------------------- | --------------- |
| Answer Faithfulness         | 回答是否忠实于证据       |
| Claim Support Rate          | 每个结论是否有证据支持     |
| Conflict Awareness          | 是否识别官方描述与用户评论冲突 |
| Target Language Consistency | 是否稳定使用目标语言      |
| Citation Accuracy           | 引用证据是否真正支持结论    |

这些指标是必要的，因为多语言 RAG 的失败不仅发生在检索阶段，也可能发生在生成阶段 [1]。

---

# 8. 消融实验设计

为了证明每个模块的作用，可以设计如下消融实验：

| 实验设置                                 | 描述             | 预期问题                |
| ------------------------------------ | -------------- | ------------------- |
| Dense-only mRAG                      | 只使用多语言向量检索     | 对精确型号和参数不稳定         |
| Hybrid mRAG                          | dense + BM25   | 精确匹配更好，但仍可能语言偏置     |
| Hybrid + Global Rerank               | 全局候选统一重排       | 排序更好，但少数语言证据可能被压制   |
| Hybrid + Language-Balanced Retrieval | 加入分语言候选池       | 跨语言覆盖更好             |
| Hybrid + Query Augmentation          | 加入多语言 query 增强 | query-document 对齐更好 |
| Full LAEP-mRAG                       | 使用完整方法         | 预期效果最好              |

重点不是只看最终答案是否正确，而是要看：

```text
系统是否同时召回了官方商品证据和真实用户评论证据。
```

# 9. 方法贡献

本文方法的主要贡献可以概括为四点。

## 9.1 语言感知的多阶段检索框架

不同于普通 mRAG 只使用一个全局 top-k，LAEP-mRAG 显式构建不同语言和不同证据类型的候选池，从而减少早期检索阶段的语言偏置。

## 9.2 面向跨语言偏置的候选生成机制

方法结合：

```text
global retrieval
language-specific retrieval
query translation
query augmentation
```

让不同语言证据都有机会进入候选池，而不是被单一排序机制提前过滤掉。

## 9.3 证据保留式生成

系统使用归一化文本帮助模型推理，但同时保留原文证据用于引用和审计。这比直接使用翻译文本生成更适合需要解释性的商品分析任务。

## 9.4 面向商品评论分析的证据建模

方法区分：

```text
official claim
user experience
negative feedback
repeated complaint
conflicting evidence
```

这使系统不仅能回答“商品好不好”，还可以解释：

```text
官方怎么说？
用户怎么说？
两者是否一致？
问题是否普遍？
```

# 10. 总结

本文提出的 LAEP-mRAG 方法可以总结为：

```text
原文多语言数据
        ↓
保留原文和语言元数据
        ↓
多语言 dense + sparse 混合检索
        ↓
查询翻译与增强
        ↓
语言平衡候选生成
        ↓
证据归一化
        ↓
多语言 / 统一语言重排
        ↓
证据保留式生成
        ↓
claim-level attribution
```

与普通多语言 RAG 相比，该方法的区别如下：

| 普通 mRAG                  | LAEP-mRAG                        |
| ------------------------ | -------------------------------- |
| 一个多语言 embedding，全局 top-k | 分语言、分证据类型生成候选                    |
| 主要依赖向量相似度                | dense + sparse + reranker        |
| 容易被英文或 query 语言支配        | 显式加入 language-balanced retrieval |
| 混合语言上下文直接输入 LLM          | 使用归一化证据辅助推理                      |
| 引用和解释能力弱                 | 保留原文证据并做 claim-level attribution |

因此，本文方法的核心观点是：

```text
多语言 RAG 的关键不是简单翻译，也不是单纯共享向量空间，
而是：语言感知的候选生成 + 去偏排序 + 证据保留式生成。
```

该研究指出，多语言 RAG 检索器可能偏好高资源语言或查询语言，但这种偏好并不一定提升最终生成质量。

该研究比较了 tRAG、MultiRAG 和 CrossRAG，说明检索后进行语言归一化有助于提升生成稳定性。

[8] Jina AI. “jina-reranker-v3.” 2025.
该模型被设计为 multilingual document reranker，可用于多语言文档重排。

[9] Cohere Documentation. “Rerank Overview.”
Cohere rerank models 支持多语言重排能力，可用于跨语言候选文档排序。
