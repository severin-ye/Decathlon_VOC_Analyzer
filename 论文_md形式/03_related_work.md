# 3 相关工作

## 3.1 评论分析与 Aspect 建模

传统评论分析研究多聚焦于情感分类、方面级情感分析和主题聚类。这些方法适合回答“评论总体偏正向还是负向”以及“用户集中讨论了哪些属性”，但通常难以把评论结论与商品本体证据显式对齐。对于电商商品分析而言，Aspect 建模仍然重要，因为它能把冗长评论转化为 aspect、sentiment、opinion、evidence span 与 usage scene 等稳定对象；但如果缺少后续证据检索，系统仍会停留在“评论怎么说”的层面，而无法回答“商品证据是否支持或解释这些评价”。

## 3.2 多模态 RAG 与跨模态检索

检索增强生成的经典工作表明，将参数化生成与外部证据检索显式分离，是提升可追溯性与知识密集任务表现的一条有效路径 [1]。进一步的检索校正研究也说明，外部证据不仅可用于回答问题，还可用于约束和修订模型生成内容 [2]。在视觉侧，CLIP 式图文对齐模型证明了统一语义空间能够支持大规模图文检索，这为将商品图片视为一等证据对象提供了直接技术基础 [3]。这些研究共同支撑了本文“商品侧保留图文双路检索”的设计判断。

不过，现有多模态 RAG 工作更多面向文档问答、知识问答或开放领域助手，查询通常来自用户问题，而不是来自评论中抽象出的需求与抱怨。相比之下，本项目把多模态检索放入商品 VOC 分析链路中，检索目标不是回答任何开放问题，而是为商品优缺点、争议点与建议生成补充支持或澄清证据。

## 3.3 两阶段检索与向量库工程

从方法论上看，RAG 及其后续工作普遍采用由候选召回到高精度筛选的粗到细范式 [1,2]。这一思想并不要求系统必须绑定某一种向量库或某一个 reranker，而强调检索阶段与最终生成阶段之间需要存在清晰的对象边界。本文延续这一层次化设计，将 embedding 服务、索引后端、重排序与证据聚合显式解耦，使“问题生成 -> 图文双路召回 -> 精排 -> 证据聚合”成为可替换、可扩展的中间链路，而不是被压缩进一次性提示词中。

## 3.4 结构化输出与工作流编排

近期关于结构化输出的研究指出，面向 schema 的任务若仍依赖完全自由生成，往往会在字段一致性、类型约束与可解析性上暴露明显脆弱性 [4,5]。与此同时，显式的 reasoning-and-acting 路线进一步表明，多步状态化工作流通常比单轮提示更适合组织工具调用、外部检索与中间对象管理 [6]。这与本文的系统设计高度一致，因为评论抽取、问题生成、证据聚合与报告生成都要求输出可解析、可落盘并可供后续阶段继续消费。

## 3.5 本项目与现有工作的差异

综合来看，当前项目并不试图提出一种新的通用多模态检索算法，也不试图用单一模型端到端取代全部中间层。它的定位更接近一个面向商品 VOC 任务的系统化整合方案：保留多模态证据，利用 Aspect 级评论建模把主观体验转成可验证对象，再以问题驱动检索和证据约束生成把评论总结、图文支持和改进建议串成一个统一闭环。正是在这个意义上，本文更关注“系统层设计是否合理、是否可复现、是否支持人审与误差分析”，而非仅仅关注某个单点模型分数。
