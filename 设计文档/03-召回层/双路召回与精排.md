# 双路召回与精排

## 双路召回

系统同时执行：

- 图像召回
- 文本召回

召回结果随后按 product_id 合并，而不是停留在单条证据层面。

但双路召回之前应增加一个中间步骤：问题生成。

## 为什么不能直接对评论召回

评论原句通常存在以下问题：

- 表达口语化，缺少稳定检索词
- 一个句子里同时混合情绪、使用场景、结论和背景
- 原句与商品证据的对应关系往往不是字面重合，而是“需要进一步验证的问题”

因此系统应采用：

- 评论 -> aspect 对象
- aspect 对象 -> k 个待澄清问题
- 问题 -> 图像路 / 文本路召回

这一步本质上是在把“主观评论”改写为“证据搜索任务”。

## 问题生成层

每个 aspect 至少应生成 1 到 k 个问题，常见类型包括：

- 事实确认问题：商品文案中是否明确支持该观点
- 视觉确认问题：商品图片中是否存在对应结构或外观证据
- 冲突澄清问题：评论问题更像真实缺陷还是预期偏差

问题对象建议至少包含：

- question_id
- source_review_id
- source_aspect
- question
- rationale
- expected_evidence_routes

## 精排的必要性

对话中已经明确指出，多模态召回比纯文本召回更容易引入噪声，因此 reranker 是强烈建议的组件。

## 合并逻辑

合并后的证据记录至少应包括：

- product_id
- image_id 或 text_block_id
- source_review_id
- source_aspect
- source_question_id
- source_question
- embedding_score
- rerank_score

## 示例记录

```json
{
  "query": "商品图像或文案中是否存在能解释夏季闷感的结构或材料证据？",
  "source_review_id": "r_1024",
  "source_question_id": "q_2001",
  "retrieved": [
    {
      "product_id": "p_1001",
      "image_id": "img_4",
      "text_id": "desc_3",
      "embedding_score": 0.79,
      "rerank_score": 0.91
    }
  ]
}
```

## 中长期演进

对话里还讨论了更前沿的召回方式，主要用于未来扩展：

- 两阶段粗召回 + 精排
- page-level 到 region-level
- multi-vector late interaction
- 更细粒度 evidence packing

这些属于当前项目后期可引入能力，而不是 MVP 起点。

但即使在 MVP 阶段，也应先把“问题生成”作为固定设计约束保留下来，避免后续从评论直召回再返工。
