# 全链路可解释性设计

## 设计目标

项目要求在每一个阶段尽量保留可解释性，因此可解释性不能是附加功能，而应是系统设计约束。

## 召回阶段的可解释性

每次召回应记录：

- source_question_id
- source_question
- query 内容
- 来源 review_id
- 来源 aspect
- 命中的 image_id 或 text_block_id
- embedding_score
- rerank_score

这意味着完整链路不再是：

- review -> aspect -> retrieval

而应变成：

- review -> aspect -> question -> retrieval

问题层是后续误差分析的关键锚点，因为它能回答“召回错，是因为问题生成错，还是因为检索错”。

## 评论抽取阶段的可解释性

每个方面对象必须保留：

- evidence_span
- usage_scene
- confidence

## 聚合阶段的可解释性

综合意见生成前，应先产出中间统计对象：

- aspect_frequency
- positive_ratio
- negative_ratio
- scene_distribution
- representative_reviews

## 建议阶段的可解释性

每条建议必须能回指到：

- 哪些评论
- 哪些商品文案
- 哪些商品图像

## 结果用途

这套机制用于支持：

- 审查结论是否合理
- 分析召回错还是总结错
- 做论文中的 error analysis
- 向答辩或业务方展示证据链
