# 文档解析与结构化召回

## 引用 1

### 外链

https://docs.langchain.com/oss/python/integrations/document_loaders/opendataloader_pdf

### 对应内容整理

该文档用于说明文档解析阶段可以保留阅读顺序、表格和边界框等结构信息，而不是把一切简单平铺成纯文本。

### 本项目使用的部分

- 结构保真优先于纯文本化的思想

### 使用的原因

虽然当前项目主要是商品 HTML 页面而非 PDF，但这条资料帮助界定了“结构保留”的重要性。

### 本项目使用的思路

迁移到本项目中，就是在商品证据层保留图片类型、文案类型和来源关系，而不是只保留一段无标签文本。

## 引用 2

### 外链

https://docs.langchain.com/oss/python/langchain/retrieval

### 对应内容整理

该文档说明 Hybrid RAG 通常不只是一次检索，而包含查询预处理、检索、验证和后处理等环节。

### 本项目使用的部分

- 混合检索视角
- 召回之后仍需后处理与验证

### 使用的原因

它适合作为“为什么本项目不是单一步骤召回，而是召回、合并、精排、证据打包”的框架性依据。

### 本项目使用的思路

本项目的双路召回、product_id 合并、rerank 和证据包构造，本质上就是一种面向商品分析场景的 Hybrid RAG 实例化。

## 引用 3

### 外链

https://github.com/HKUDS/RAG-Anything/blob/main/README.md

### 对应内容整理

RAG-Anything 代表的是结构化元素保留和关系增强的多模态检索路线，强调向量相似度与图遍历结合、模态感知排序和关系一致性。

### 本项目使用的部分

- 结构化多模态召回的参考路线
- 关系增强检索的概念背景

### 使用的原因

它适合作为相关工作中“不是所有多模态系统都走纯原始模态视觉检索”的对照路线。

### 本项目使用的思路

本项目目前没有走图结构召回，但保留了对象关系层，为未来把商品图片、商品文本、评论方面对象和建议证据用关系图组织起来留了空间。

## 引用 4

### 外链

https://developer.nvidia.com/blog/how-to-build-a-document-processing-pipeline-for-rag-with-nemotron/

### 对应内容整理

该文档展示了一种更工程化的结构化多模态召回思路：

- 保留表格 markdown
- 保留图表裁剪
- 用文本、图像或图文联合 embedding
- 在 top-k 之后做 multimodal reranking

### 本项目使用的部分

- “保留原始元素 + 保留结构关系 + 再做多模态重排”的工程思路

### 使用的原因

它特别适合支撑本项目中“不是所有模态都必须原样输入模型，而是关键证据才进入最终生成”的原则。

### 本项目使用的思路

当前项目中的商品图片、商品文案、评论方面对象三者的证据打包，本质上与这种结构化保留再重排的思路一致。

### 对应内容整理

Docling 提供 PDF、HTML、图片等多种文档结构解析能力，并保留 layout、reading order、table structure、formula 和 lossless JSON。

虽然当前项目首阶段不以 PDF 为主，但未来如果纳入商品说明书、宣传册、规格附件，Docling 会是合适扩展项。
